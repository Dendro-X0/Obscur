## Goal 1: Community Modes Implementation (P0)

### Background

From `docs/rewrite/33-community-modes-and-relay-guarantees.md`:

Communities currently present one universal feature set, but runtime capabilities differ based on relay environment:

- **Public Default Relays** — Lowest friction, weakest membership/directory guarantees
- **Trusted Private Relays** — Stronger guarantees, requires operator control
- **Intranet/Operator-Controlled** — Strongest guarantees, most configuration

### Deliverables

#### 1.1 Mode Selector

**UI**: Community creation dialog shows mode selection:

```
┌─────────────────────────────────────────┐
│  Create Community                       │
│                                         │
│  [ ] Sovereign Room                     │
│      Public relays, E2EE content        │
│      ⚠️ Weak membership guarantees       │
│                                         │
│  [ ] Managed Workspace                  │
│      Private relays, E2EE content       │
│      ✓ Strong membership guarantees     │
│      (Requires trusted relay setup)     │
└─────────────────────────────────────────┘
```

**Canonical Owner**: `apps/pwa/app/features/groups/components/community-creation-dialog.tsx`

**Required Changes**:
- [ ] Add `mode` field to community metadata (CRDT)
- [ ] Update creation dialog with mode selector
- [ ] Mode-specific help text and warnings

#### 1.2 Honest Membership Visibility

**Rule**: Never claim "live" or "synced" when not true.

**Implementation**:
- [ ] "Last seen" shows actual evidence timestamp, not optimistic claim
- [ ] "Syncing..." indicator when membership ledger is converging
- [ ] "Some members may be hidden" warning for public-relay communities

**Canonical Owner**: `apps/pwa/app/features/groups/components/community-member-list.tsx`

#### 1.3 Relay Capability Indicators

**UI**: Community header shows relay mode badge:

```
Community Name                    [Sovereign Room]
                                  Relay: Public Mix
                                  Guarantees: Content Privacy ✓
                                              Membership ⚠️
```

**Canonical Owner**: `apps/pwa/app/features/groups/components/community-header.tsx`

## Goal 2: Restore & Media Convergence (P0)

### Problem Statement

From current open blockers:

> **Fresh-device restore**: media clears from message history / Vault after login

Media attachments are lost when restoring account on a new device, breaking conversation history completeness.

### Root Cause Analysis

Likely causes:
1. Vault restore runs before media metadata sync completes
2. Message references restored but media blobs not fetched
3. Race condition between restore merge and vault initialization

### Deliverables

#### 2.1 Vault Restore Ordering

**Required**: Media metadata must restore before vault becomes active.

**Changes**:
- [ ] Ensure `encrypted-account-backup-service.ts` restores vault namespace before activation
- [ ] Add explicit dependency: vault metadata → vault blob fetch → vault ready

**Canonical Owner**: `apps/pwa/app/features/account-sync/services/encrypted-account-backup-service.ts`

#### 2.2 Media Blob Recovery

**Required**: Missing media blobs should be re-fetched from CAS (Content-Addressed Storage).

**Changes**:
- [ ] Detect orphaned media references (message has hash, vault missing blob)
- [ ] Background fetch from CAS by content hash
- [ ] Progress indicator for media recovery

**Canonical Owner**: `apps/pwa/app/features/vault/services/cas-media-recovery.ts` (new)

#### 2.3 Restore Verification

**Required**: Post-restore health check confirms media completeness.

**Changes**:
- [ ] After restore, scan for media reference orphans
- [ ] Report: "X media items recovered" or "Y items unavailable"
- [ ] Manual re-check button in Settings → Storage

**Canonical Owner**: `apps/pwa/app/features/vault/components/vault-health-panel.tsx` (new)

### Success Criteria

- [ ] Fresh-device restore preserves 100% of Vault media
- [ ] DM history shows all media thumbnails post-restore
- [ ] No "missing image" placeholders in restored conversations

## Goal 3: Realtime Voice Reliability (P1)

### Background

Ghost calls from historical invites during account sync were mitigated in v1.4.4 with TTL-based expiration, but edge cases may remain.

### Deliverables

#### 3.1 Session Single-Flight

**Required**: Only one active call session per conversation at any time.

**Changes**:
- [ ] Enforce single-flight in `voice-call-controller.ts`
- [ ] Reject new invite if existing session active
- [ ] Clear stale sessions on controller mount

**Canonical Owner**: `apps/pwa/app/features/voice/services/voice-call-controller.ts`

#### 3.2 Degraded Connection Handling

**Required**: Explicit degraded/unsupported outcomes, not silent failures.

**Changes**:
- [ ] Detect relay voice capability (NIP-103 support)
- [ ] Show "Voice calls may be unreliable on current relay" warning
- [ ] Fallback to voice notes when voice calls unsupported

**Canonical Owner**: `apps/pwa/app/features/voice/components/voice-call-button.tsx`

#### 3.3 Ghost Call Verification

**Required**: Confirm TTL-based expiration works in production.

**Verification**:
- [ ] Test: 6-minute-old invite during sync → should NOT trigger call UI
- [ ] Test: 2-minute-old invite during sync → SHOULD trigger call UI (live call scenario)

**Test File**: `apps/pwa/app/features/voice/services/realtime-voice-history-replay-policy.test.ts`

## Goal 4: Security Feature Production Wiring (P1)

### Background

v1.4.6 created security services but they use placeholder/demo data:

- `KeyChangeDetector` — tracks known keys but not wired to contact changes
- `RelayTrustScorer` — calculates scores but not wired to live relay metrics
- `SecurityAuditLog` — accepts events but no publishers call it

### Deliverables

#### 4.1 Key Change Detection Live

**Required**: Detect actual contact public key changes and alert user.

**Changes**:
- [ ] Wire `KeyChangeDetector.trackContactKey()` when contact profile loaded
- [ ] Check for key mismatch on each profile refresh
- [ ] Show security alert banner in conversation when key changes
- [ ] Require re-verification before sending messages to changed identity

**Canonical Owners**:
- Detection: `apps/pwa/app/features/network/hooks/use-contact-profile.ts`
- Alert UI: `apps/pwa/app/features/messaging/components/key-change-alert.tsx` (new)

#### 4.2 Relay Trust Scoring Live

**Required**: Calculate trust scores from actual relay behavior.

**Changes**:
- [ ] Record delivery success/failure per relay per message
- [ ] Track latency samples per relay
- [ ] Calculate rolling 24h health score
- [ ] Update trust level (high/medium/low/untrusted) based on thresholds

**Canonical Owner**: `apps/pwa/app/features/relays/hooks/enhanced-relay-pool.ts`

**Metrics**:
- Delivery rate: >95% = high, 80-95% = medium, <80% = low
- Latency: <200ms = bonus, >1000ms = penalty

#### 4.3 Security Audit Log Live

**Required**: Log real security events as they occur.

**Changes**:
- [ ] `SecurityAuditLog.logEvent()` calls from:
  - Identity unlock/lock
  - Key verification actions
  - Key change detections
  - Relay trust level changes
  - Failed message deliveries (suspicious patterns)
- [ ] Settings → Security → Audit Log viewer

**Canonical Owners**:
- Publisher: `apps/pwa/app/features/security/services/security-event-publisher.ts` (new)
- Viewer: `apps/pwa/app/features/settings/components/security-audit-log-viewer.tsx` (new)
