# v1.4.0 In-Place Rewrite and Resilience Plan

Last Updated (UTC): 2026-04-21  
Status: Active  
Release Lane: `v1.4.0`

## Purpose

This file is the single canonical roadmap and execution contract for `v1.4.0`.

`v1.4.0` is not a feature-trim or MVP restart lane.

It is the in-place community-system overhaul and resilience lane whose goal is:

1. complete a major overhaul of the community systems,
2. validate the supported community features end to end before release,
3. keep the underlying architecture scalable, modular, and maintainable,
4. preserve adjacent core trust paths while that overhaul lands.

## Deletion Guard (Do Not Remove Early)

This file must remain in the repository and must not be deleted until every item below is true:

1. all tasks in `Execution Checklist` are complete,

## Why v1.4.0 Exists

From a user-experience standpoint, the current state is still unacceptable:

1. fresh-window/fresh-device login can restore only part of DM history,
2. current community behavior still over-promises on membership and directory truth,
3. fixes in one lane can still destabilize adjacent trust paths,
4. the architecture remains structurally modular but behaviorally fragile.

This release exists to overhaul the community systems into an honest,
relay-capability-aware product while preserving the privacy-first model.

## Product Rules for This Lane

### Non-Negotiable

`v1.4.0` does not mean:

1. restarting the MVP,
2. disabling communities or restore,
3. reducing the product to a simpler centralized demo,
4. hiding instability behind feature flags without naming canonical owners.

### Required

`v1.4.0` does mean:

1. rewrite in place,
2. one canonical owner per fragile truth surface,
3. convert compatibility bridges into named, measurable debt,
4. add ratchet tests and diagnostics before expanding again,
5. judge progress by runtime continuity, not local patch success.
6. permit destructive owner replacement when repeated regressions prove the
   current module cannot be salvaged safely.

## Aggressive Strategy Rule

The project should now assume that some current owners are replacement-only.

For any lane where:

1. the same regression class has resurfaced repeatedly,
2. runtime UX remains broken after multiple bounded fixes,
3. new fixes keep reopening adjacent failures,

the correct action is:

1. stop patching that owner,
2. document the blocker explicitly,
3. delete or quarantine the failing owner module behind a minimal compatibility
   shell,
4. rebuild the owner from contracts and deterministic boundaries upward.

This is especially relevant for:

1. restore/import ownership,
2. DM read-model ownership,
3. community membership/directory ownership.

## Scope Summary

### In Scope

1. community modes and relay guarantees,
2. community membership and roster projection authority,
3. room-key, sendability, and community coordination contracts,
4. restore import ownership where it affects community continuity,
5. DM conversation and timeline authority where it supports community continuity,
6. relay transport evidence and fault-tolerance contracts,
7. modularization and anti-drift test ratchets for fragile lanes,
8. runtime resilience under limited contributor/context conditions.

### Out of Scope

1. removing major user-facing features,
2. rewriting the entire visual design system,
3. replacing the decentralized/local-first model,
4. introducing a backend that requires private plaintext ownership or removes
   user portability.

Allowed:

1. backend-authoritative coordination for session, restore, membership,
   discovery, and indexing metadata,
2. operator-hosted or self-hosted services that prioritize maintainability and
   scaling over protocol purity.

## Canonical v1.4.0 Goals

### Goal 1. Community System Overhaul

The community system must ship as an explicitly-modeled product, not a vague
"groups" feature set.

Completion means:

1. community modes are explicit,
2. user-facing guarantees match relay/runtime capability,
3. unsupported community promises are removed or demoted,
4. the architecture for community planes is modular and iterable.

### Goal 2. One Community Membership Projection

Community UI must read one projection owner instead of mixed heuristics.

Completion means:

1. member list and group list agree where the chosen mode promises them,
2. join/leave status updates in real time where the configured relay mode supports them,
3. restored membership includes other known members when evidence exists,
4. invite response cards are not the only surviving proof of membership.

### Goal 3. Relay Capability and Guarantee Split

Community features must follow the relay/runtime environment the user actually
configured.

Completion means:

1. public/default relays map to sovereign-room guarantees,
2. stronger workspace guarantees are only offered when stronger relay/runtime assumptions are explicit,
3. Settings and community UX expose those tradeoffs clearly,
4. the default community creation path stays low-friction for non-technical users while advanced users can opt into stronger relay-backed modes.

### Goal 4. Restore and Supporting Truth Convergence

Users logging in on a new window/device should recover the supporting DM and
community state required by the chosen community mode.

Completion means:

1. restore no longer thins DM history in supporting lanes,
2. restore no longer collapses joined communities to self-only,
3. direct chat-state restore is compatibility-only and bounded,
4. imported state survives subsequent projection/provider refreshes.

### Goal 5. Transport and Runtime Resilience

Runtime/relay behavior must not destabilize core product flows.

Completion means:

1. route liveness remains responsive under runtime churn,
2. publish and subscribe evidence are diagnosable,
3. degraded states fail visibly and safely instead of silently thinning truth.

### Goal 6. Anti-Drift Iteration Discipline

Future updates should be less likely to re-break the same trust paths.

Completion means:

1. each module lane has ratchet tests,
2. compatibility bridges are named and measured,
3. owner docs and handoffs stay current,
4. future slices remove overlap rather than adding new overlap.

Goals:

1. unify joined roster recovery,
2. preserve real-time member status updates where supported,
3. converge network/community page/member-modal truth.

Goals:

1. align membership truth with room-key/sendability truth,
2. define which people/directory features are stable versus best-effort,
3. support coordination features that match the configured relay mode.

### D. Restore and Account Convergence

Goals:

1. canonicalize backup restore import,
2. preserve DM/private history across new-window and fresh-device login,
3. make restore replay and live catch-up non-competing.

### E. DM Authority and History Integrity

Goals:

1. unify DM list/timeline truth,
2. preserve delete and attachment durability,
3. stop thinner-after-restore outcomes.

### F. Transport and Runtime Protection

Goals:

1. keep runtime and relay owners diagnosable,
2. keep page-switch and degraded behavior bounded,
3. protect core flows from background churn.

### G. Modularization and Test Ratchets

Goals:

1. split fragile lanes by contract/import/projection/persistence/UI adapter,
2. enforce the module test ladder,
3. make each slice safer to iterate next time.

1. identify the canonical owner,

3. avoid multi-owner feature patches.

1. move providers/pages to canonical projections,

2. confirm diagnostics match the owner story,

4. only then call the lane stable.

## Acceptance Criteria for v1.4.0

`v1.4.0` is only successful when all are true:

1. community modes and guarantees are explicit and reflected in UI/settings/docs,
2. sovereign-room mode works reliably on public/default relays without false claims,
3. joined communities reconstruct with the correct known members or bounded best-effort outputs according to mode guarantees,
4. supported community join/leave/directory behaviors validate against the configured relay mode,
5. restored state is not later demoted by provider hydration or canonical append ordering,
6. the next engineering slices have stronger anti-drift gates than the current codebase.

## Risks

1. Risk: ad hoc compatibility bridges remain active long enough to undermine canonical owners.
   Mitigation: make every bridge measurable and explicitly temporary.

2. Risk: tests stay green while runtime truth is still thinner.
   Mitigation: runtime replay is part of done for trust-sensitive lanes.

3. Risk: contributors keep solving surface bugs instead of owner overlap.
   Mitigation: this roadmap requires owner identification before implementation.

4. Risk: rewrite pressure gets interpreted as scope reduction.
   Mitigation: keep feature-complete product surface as a lane invariant.
