## Scope Summary

### In Scope

1. keep most UI components/assets available locally and usable offline,
2. preserve network-dependent flows only where required (relay state, message sync, publish),
3. add desktop in-app streaming update flow with cryptographic verification,
4. keep core messaging/nav reliability from regressing.

### Out Of Scope (This Lane)

1. removing all network requirements from messaging/relay features,
2. replacing release infrastructure outside updater needs,
3. broad design-system rewrite unrelated to offline/update goals.

## Product Principles for This Lane

1. offline UI is a product contract, not a best-effort cache.
2. network truth remains evidence-backed for messaging/sync state.
3. one canonical owner per lifecycle path (no duplicate update/offline owners).
4. unsupported updater/runtime states must fail visibly and safely.

## v1.3.7 Baseline Summary (Starting Point)

1. Blank timeline + `Load More` regressions were reduced via adaptive hydration, hidden-row filtering, and virtualizer recovery.
2. Sidebar click-drop regressions were reduced by explicit router navigation requests with fallback watchdog.
3. Remaining fragile surfaces for this lane:
   - fresh-device restore blankness perception,
   - account-sync warmup clarity in empty states,
   - offline UX boundaries not yet formalized,
   - in-app streaming update owner path not yet implemented.

### A. Stability Carry-Over (v1.3.7 hardening continuity)

Goal: prevent recurrence of blank timeline + `Load More` friction and intermittent sidebar navigation drops.

Deliverables:

1. runtime replay evidence for blank-history and nav stress paths (desktop + PWA),
2. diagnostics capture guide for remaining edge failures,
3. focused regression tests added where gaps still exist,
4. file upload/processing progress UI no longer stalls at `0%` during active work.

Exit Signals:

1. no blank timeline when visible rows exist in latest window,
2. sidebar route clicks settle deterministically under active chat load,
3. diagnostics events available when failures are intentionally induced.

### B. Hybrid Offline-First UI Foundation

Goal: make core app shell/UI largely usable offline without relay/network dependency.

Deliverables:

1. inventory of all UI asset/component network dependencies,
2. local-first asset policy (fonts/icons/images/ui libs) with CI guard against accidental remote asset regressions,
3. service worker/app-shell caching strategy for deterministic offline boot,
4. explicit boundary map of offline-capable vs network-dependent surfaces,
5. UX copy/states for offline mode and reconnection.

Exit Signals:

1. app shell and core navigation render offline from local assets only,
2. network-only modules degrade gracefully with clear status,
3. offline replay pass documented with deterministic steps and evidence.

### C. In-App Streaming Updates (Desktop)

Goal: users can update inside the app without visiting GitHub Releases or website download pages.

Deliverables:

1. signed release manifest contract (version/channel/platform artifact/checksum/signature),
2. updater runtime owner in desktop app:
   - check available version,
   - download in background with progress,
   - checksum/signature verification,
   - apply/restart flow,
3. staged rollout controls (channel, rollout %, kill switch, min-safe version),
4. user-facing update UX in settings/system surfaces (notes, progress, error states, retry),
5. CI/release pipeline step to publish manifests and artifacts atomically.

Exit Signals:

1. upgrade from prior stable build to target build can complete fully in-app,
2. invalid checksum/signature blocks install and preserves current version,
3. disabled rollout/kill switch prevents unsafe updates deterministically.

### D. Security and Recovery Gates

Goal: updater and offline caches remain integrity-safe and recoverable.

Deliverables:

1. signed metadata verification path with explicit failure reasons,
2. updater rollback/fallback behavior contract,
3. offline cache invalidation/versioning strategy for broken assets,
4. incident playbook for failed update rollout.

Exit Signals:

1. tampered update payload is rejected with clear user-visible state,
2. failed update returns to known-good current build path,
3. operations checklist exists for emergency rollback.

## Risks and Mitigations

1. Risk: offline cache serves stale/broken UI bundle.
   Mitigation: versioned cache keys + controlled invalidation + fallback to safe local shell.
2. Risk: updater introduces second lifecycle owner and race with runtime boot.
   Mitigation: one updater runtime owner with explicit state machine.
3. Risk: rollout mistake ships broken artifact broadly.
   Mitigation: staged rollout controls + kill switch + signed manifest verification.
4. Risk: regressions reintroduce blank timeline/nav stalls.
   Mitigation: preserve existing diagnostics and replay these paths in every release lane.
