## v1.4.4: Anti-Abuse Foundations

**Goal**: Prevent bot account creation while preserving privacy-first, no-phone/email philosophy.

### Account Creation Checkpoints

Since Obscur uses private-key identity without phone/email verification, we need alternative "proof of humanity" checkpoints:

#### 1. Proof-of-Work Key Generation (Client-Side)
- **Mechanism**: Require iterative key generation with hash prefix requirement
- **UX**: "Securing your identity..." progress indicator during account creation
- **Cost**: ~2-5 seconds of computation on modern devices
- **Effect**: Mass account creation becomes computationally expensive

```
Implementation: apps/pwa/app/features/auth/services/pow-key-generator.ts
Target: 2-5 second generation time
Prefix: Require 4-6 zero bits in public key hash
```

#### 2. Relay-Based Invitation Codes (Optional)
- **Mechanism**: Existing users can generate limited-use invite codes
- **UX**: "Have an invite code?" option during onboarding
- **Effect**: Organic growth through trusted network edges
- **Rate limit**: 5 invites per user per month default

#### 3. CAPTCHA Checkpoint (Minimal)
- **Mechanism**: hCaptcha or Cloudflare Turnstile for signup flow
- **UX**: Single "Verify you're human" step
- **Privacy**: No personal data linked to identity
- **Fallback**: Proof-of-work option for privacy-maximalists

#### 4. Account Warm-Up Period
- **Mechanism**: New accounts have limited actions for first 24 hours
- **Limits**: 
  - Max 10 connection requests per hour
  - Max 3 community joins per hour
  - Cannot create communities until account age > 7 days
- **Effect**: Bot accounts have limited blast radius even if created

### Implementation Plan

| Checkpoint | Owner | Est. Effort | Priority |
|------------|-------|-------------|----------|
| PoW Key Generation | Auth service | 3 days | P0 |
| Invite Code System | Relay gateway | 4 days | P1 |
| CAPTCHA Integration | Auth UI | 2 days | P1 |
| Account Warm-Up | Rate limiter | 2 days | P0 |

## v1.4.5: Security Hardening

**Goal**: Strengthen identity verification, relay trust, and audit capabilities.

### Key Verification Improvements

#### Cross-Device Key Verification
- **Mechanism**: Display visual fingerprint of public key for manual verification
- **UX**: "Verify Identity" button showing identicon + key fragment
- **Use case**: High-trust connections (journalists, activists) can verify keys out-of-band

#### Key Change Detection
- **Mechanism**: Alert when contact's identity key changes unexpectedly
- **UX**: "Security Alert" banner in conversation
- **Action**: Require re-verification before sending messages

### Relay Trust Scoring

#### Dynamic Relay Reputation
- **Mechanism**: Track relay behavior metrics locally
- **Metrics**:
  - Message delivery success rate
  - Response latency
  - Uptime percentage
  - Spam/blocking reports from user
- **UX**: Subtle indicator (green/yellow/red dot) next to relay in settings

#### Automatic Relay Fallback
- **Mechanism**: When primary relay fails, auto-promote healthy backup
- **UX**: Brief "Switching to backup relay..." toast
- **Config**: User can disable auto-switch in advanced settings

### Audit Logging

#### Local Security Log
- **Mechanism**: Append-only log of security-relevant events
- **Events**:
  - Identity key changes
  - Suspicious connection requests
  - Relay failures/switches
  - Unusual message patterns
- **Location**: `~/.obscur/security-audit.log` (encrypted with user key)
- **UX**: "Security History" panel in settings for review

## Cross-Cutting Concerns

### Privacy Preservation

All security measures must preserve Obscur's core privacy guarantees:

1. **No phone/email linking** - Never require PII for account creation
2. **No centralized identity** - No server-side user database
3. **Local-first** - Anti-abuse data stays on device where possible
4. **Transparent** - Users can see what security data is collected

### UX Principles

1. **Progressive disclosure** - Advanced security features hidden by default
2. **Honest state** - Never claim "live" or "synced" when not true
3. **Graceful degradation** - Security checks fail open, not closed
4. **Minimal friction** - Legitimate users should rarely notice protections

### Test Requirements

Each release requires:

## Anti-Bot Strategy: Detailed Explanation

The challenge: Obscur has no central server, no phone verification, and no email confirmation. This is a feature for privacy, but creates vulnerability to bot farms.

### Threat Model

**Attacker capabilities**:
- Can run JavaScript in browsers/headless Chrome
- Has moderate compute resources (can solve CAPTCHAs)
- Wants to create 1000s of accounts for spam/scam

**Our constraints**:
- Cannot add phone/email verification (violates privacy)
- Cannot require payment (excludes users)
- Cannot use centralized identity providers (defeats purpose)

### Defense Layers

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: Economic Barrier (PoW Key Generation)         │
│  - Cost: 2-5s per account × 1000 accounts = ~1hr compute  │
│  - Attacker must invest GPU time for bulk creation       │
├─────────────────────────────────────────────────────────┤
│  Layer 2: Rate Limiting (Warm-Up Period)                  │
│  - New accounts throttled for 24h                         │
│  - Limits blast radius if attacker persists             │
├─────────────────────────────────────────────────────────┤
│  Layer 3: Social Graph (Invite Codes)                     │
│  - Organic growth through trusted edges                   │
│  - Attacker must infiltrate real user networks          │
├─────────────────────────────────────────────────────────┤
│  Layer 4: Behavioral (CAPTCHA)                            │
│  - Fallback for privacy-maximalists who skip PoW        │
│  - Standard bot detection for browser automation        │
└─────────────────────────────────────────────────────────┘
```

### Why This Works

**Legitimate users**: Experience 2-5s delay during account creation, then normal usage.

**Bot farms**: Face economic disincentive (compute cost) + throttled impact even if they persist.

**Privacy preserved**: No phone, no email, no centralized database. All checks local or optional.
