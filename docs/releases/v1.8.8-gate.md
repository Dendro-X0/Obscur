## CI vs full release (important)

Two different automation paths run on GitHub:

| Trigger | Workflow | What it does |
|---------|----------|--------------|
| **Push to `main`** | `docs-check.yml`, `reliability-gates.yml`, `version-check.yml` (path-filtered) | Fast gates on commits — **not** installer builds |
| **Push tag `v*`** | [`.github/workflows/release.yml`](../../.github/workflows/release.yml) — **Obscur Full Release** | Preflight → desktop (3 OS) → Android → web bundle → verify → **publish installers** |

Green checks on a **commit push** (docs-check + reliability-gates only) do **not** mean the full release ran.

A GitHub Release that shows **only** “Source code (zip/tar.gz)” was created manually or before `publish-release` — those archives are auto-generated by GitHub, not our pipeline.

### Likely cause for v1.8.8

1. Tag `v1.8.8` pointed at a commit **before** the docs-check fix (stale-path-ref to local env file in gate doc prose) → **Obscur Full Release** failed in `preflight-checks` / `pnpm docs:check`.
2. A GitHub Release page was published with source-only assets while the full workflow was still failing or was skipped.

### Recovery (historical — use v1.8.9 instead)

**Do not retag `v1.8.8`.** Ship [v1.8.9-gate.md](./v1.8.9-gate.md) on current `main`.

### Transient Linux desktop build (`curl` / `Transferred a partial file`)

If **Build Desktop (ubuntu-latest)** fails while fetching crates (e.g. `futures-macro`), it is usually a **flaky crates.io download**, not an app defect. Windows/macOS often pass on the same run.

1. **Re-run** the failed job: Actions → failed **Obscur Full Release** run → **Build Desktop (ubuntu-latest)** → Re-run failed jobs.
2. After merging workflow hardening: `release.yml` uses Cargo cache, `CARGO_NET_RETRY`, and retry loops on `cargo fetch` / `tauri build`.
