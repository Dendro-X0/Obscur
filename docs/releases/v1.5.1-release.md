# v1.5.1 Release Notes

**Codename:** Honest Hide  
**Status:** Release candidate — tag `v1.5.1` pending manual sign-off  
**Builds on:** [v1.5.0](./v1.5.0-release.md) (Gateway & Membership)  
**Target:** Patch release focused on DM visibility trust

---

## Summary

v1.5.1 replaces the **misleading “delete” experience** with **honest, durable local hiding** on the v1.5.0 ClientGateway stack. It restores the *user trust* that existed in v1.3.15 for session-local removal (messages stay gone from the thread you are viewing) without claiming global erasure on Nostr.

**Not in v1.5.1 UI:** **Recall for everyone** — removed; Nostr cannot guarantee cross-client unsend. Supported action: **Hide on this device** only. See [messaging/dm-recall-for-everyone.md](../messaging/dm-recall-for-everyone.md).

Execution scope and gates: [program/v1.5.1-scope.md](../program/v1.5.1-scope.md).

---

## Problem statement

After v1.5.0, users still hit behaviors that feel like broken deletes:

| Symptom | Why it erodes trust |
|---------|---------------------|
| Hidden messages **reappear** after refresh or account restore | Multiple stores replay rows without a single suppression contract |
| Copy says **“Delete”** / **“Delete for everyone”** | Implies server-grade erasure; Nostr only supports cooperative hide |
| v1.3.15 felt more reliable for “gone from my view” | Simpler read path: filter in-memory list + tombstone before replay |

v1.5.1 does **not** port v1.3.15 code (use git tag **`v1.3.15`** for historical read-only reference). It applies the same **invariants** through ClientGateway + hydrate authority landed in v1.5.0.

---

## Highlights (planned)

### A — Durable concealment (“Hide on this device”)

- Rename user-facing actions from **Delete for me** → **Hide on this device** (and batch equivalents).
- One canonical write path: suppression / tombstone + projection `DM_REMOVED` where applicable.
- One canonical read path: hydrate + live overlay respect suppression (no parallel filter in UI).
- **Exit trust test:** hide → refresh → still hidden; hide → account restore → still hidden.

### B — Reversal (“Show again on this device”)

- Intentional **undo** for local hides (clear suppression for chosen identities).
- Replaces the bad pattern where messages **unexpectedly resurrect** after sync.

### C — No recall UI

- **Recall for everyone** is not offered in menus (not ecosystem-guaranteed on Nostr).
- Peers using Obscur may still send delete commands; ingress may apply cooperative hide when understood.

## User-visible changes (when shipped)

| Before (v1.5.0) | After (v1.5.1) |
|-----------------|----------------|
| “Delete for me” | **“Hide on this device”** |
| “Delete for everyone” / “Recall for everyone” | **Removed** — use **Hide on this device** |
| Silent reappearance after restore | Hidden rows stay hidden; optional **Show again** entry |

Short help note (once): *Hiding affects this device. Other devices may still show a message until account sync converges. Hiding is not the same as erasing events on the network.*

---

## Known limitations (expected at ship)

| Area | Limitation |
|------|------------|
| Network erasure | Peers on non-Obscur clients or pre-fetch copies may retain plaintext |
| Cross-client unsend | Not a product feature; other Nostr clients are unaffected by local hide |
| Cross-device | Hide intent should converge via account-sync; may lag — surface status, not “deleted” |
| Mobile store installers | Not published on GitHub Releases (unchanged from v1.5.x policy) |

Accepted limitation context: [messaging/deletion-roster-limitations.md](../messaging/deletion-roster-limitations.md).

## Verification (release gate preview)

Automated and client gates are defined in [v1.5.1-durable-hide-scope-and-gate.md](./v1.5.1-durable-hide-scope-and-gate.md).

Minimum **manual client testing** before tag:

1. Web + Desktop, same profile: hide DM → refresh → still hidden on both (when sync enabled).
2. Hide DM → trigger account restore / rehydrate → still hidden.
3. **Show again** restores visibility on that device only.
4. Incoming messages and send path unchanged (smoke).
5. Batch hide mode uses updated copy; no enabled “Delete for everyone” affordance.
