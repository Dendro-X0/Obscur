# v0.9 Security + Identity Restructure Roadmap

_Last reviewed: 2026-03-07 (baseline commit ab08104)._

## Summary

This roadmap is a hard prerequisite for broad platform expansion (especially community systems).

Primary outcome:

1. Stable decentralized messaging with verifiable E2EE behavior.
2. Identity model that is secure, recoverable, and usable by daily users.
3. Deterministic delivery/sync/storage guarantees under relay instability.

## Locked Decisions

1. Security and identity restructuring is the next major priority.
2. No financial features (no wallet/payments/trading) in Obscur.
3. Keep decentralized architecture and relay interoperability.
4. Global search and social discovery cannot be required for core friend-add and messaging.
5. Community system expansion starts only after security/identity gates are green.

## Architecture Direction

1. **Protocol core first**:
- Move security-critical logic to a shared protocol core (Rust-first preferred).
- Keep UI stack decoupled from cryptographic/session correctness.
2. **Identity separation**:
- Root identity key (recovery/offline-sensitive).
- Per-device/session keys (operational day-to-day).
- Device authorization/revocation events.
3. **Transport resilience**:
- Relay scoring, circuit-breaker behavior, quorum publish, deterministic outbox states.
4. **Storage trust model**:
- Local encrypted durable state + untrusted remote encrypted blobs/events.

## Stack and Framework Position

1. Do not rewrite frontend first.
2. Add/expand shared protocol core (Rust + audited crypto/session state machine).
3. Keep current app shells (desktop/web-dev/mobile) using adapters over protocol core APIs.
4. Any framework changes are secondary and only justified by measurable delivery gains.

## Ticketized Roadmap (Execution Order)

### Wave 1 — Security + Identity Foundation (P0)

1. `V090-SEC-001` Threat Model and Security Invariants Refresh
- Update attacker model, trust boundaries, key compromise assumptions, and relay adversary behaviors.
- Freeze security invariants used for go/no-go checks.

2. `V090-ID-001` Identity Domain Model Split
- Define root key, device key, session key responsibilities and lifecycle.
- Enforce single active unlocked profile/session invariant.

3. `V090-ID-002` Device Authorization + Revocation Events
- Add signed device authorization records.
- Add deterministic revocation flow and propagation checks.

4. `V090-ID-003` Key Reveal/Export Reliability Contract
- Define policy-gated reveal path with deterministic fallback behavior.
- Eliminate runtime-specific dead ends (biometric unavailable, native command unavailable).

5. `V090-SEC-002` Cryptographic Primitive Review
- Confirm algorithm set, parameter choices, nonce/entropy handling, and key derivation policy.
- Publish implementation-level crypto checklist with test vectors.

### Wave 2 — E2EE Protocol Hardening (P0)

6. `V090-E2EE-001` Direct Message Protocol Upgrade
- Implement or harden X3DH + Double Ratchet style session lifecycle.
- Add replay protection and deterministic session recovery rules.

7. `V090-E2EE-002` Metadata Minimization
- Reduce leaked metadata in transport and storage paths where protocol-compatible.
- Explicitly classify unavoidable metadata exposure.

8. `V090-E2EE-003` Message Verification and Tamper Detection
- Strong message authenticity checks and signature/path diagnostics.
- Deterministic reject reasons exposed in diagnostics.

### Wave 3 — Relay and Delivery Reliability (P0)

9. `V090-RELAY-001` Relay Circuit Controller
- Formalize relay states (`healthy`, `degraded`, `cooling_down`, `unavailable`) and transitions.
- Suppress reconnect storms and duplicate subscriptions.

10. `V090-RELAY-002` Outbox and Quorum Delivery Contract
- Authoritative outbox statuses:
`queued -> publishing -> sent_partial|sent_quorum -> accepted|rejected|expired|failed`.
- Bounded retry/backoff with duplicate suppression.

11. `V090-SYNC-001` Checkpoint/Backfill Correctness
- Deterministic timeline checkpoints and targeted backfill repair.
- No broad resync unless repair path fails.

### Wave 4 — Hybrid Storage Reliability (P0)

12. `V090-STOR-001` Local Encrypted Store Baseline
- Define canonical local database model and encryption-at-rest path.
- Add startup health check and recovery reporting.

13. `V090-STOR-002` Remote Blob/Event Separation
- Enforce plaintext never leaves trusted local path for protected media/messages.
- Remote providers store ciphertext blobs and envelope metadata only.

14. `V090-STOR-003` Recovery UX and Diagnostics
- One-click non-destructive rebuild paths with deterministic reporting.
- Local-only counters and user-facing reason codes.

### Wave 5 — UX Safety and Developer Ergonomics (P1)

15. `V090-UX-001` Security-Grade UX Baselines
- Make critical flows legible and deterministic:
identity reveal, export, friend-add, request status, publish status.
- Remove ambiguous loaders and silent failures.

16. `V090-UX-002` Friend Add Without Long Keys
- Canonical contact card + link + QR.
- Optional short alias/index resolver as convenience layer, never cryptographic source of truth.

17. `V090-DEV-001` Web-dev/desktop parity harness
- Preserve two-user testing parity and deterministic behavior across runtimes.

## Public/Internal Interface Contracts

1. `IdentityRootState`, `DeviceKeyRecord`, `SessionKeyState`.
2. `MessageSessionState`, `E2EEHandshakeResult`, `MessageVerifyResult`.
3. `RelayCircuitState`, `QuorumPublishReport`, `OutboxTransitionResult`.
4. `StorageHealthState`, `StorageRecoveryReport`, `EncryptedBlobDescriptor`.
5. `SecurityReasonCode` set for all user-impacting failures.

No protocol-breaking remote schema migration without explicit compatibility plan.

## Quality Gates and Testing

### Unit (Mandatory)

1. Crypto test vectors (key derivation, signing, encryption/decryption, ratchet transitions).
2. Identity/device/session transition invariants.
3. Outbox retry/backoff/cooldown determinism.
4. Checkpoint repair and dedupe/order idempotency.

### Integration (Mandatory)

1. Desktop A + Web-dev B add/request/accept/DM under relay churn.
2. Session restart and key reveal/export behavior across supported runtimes.
3. Relay partial outage with quorum success and explicit partial diagnostics.
4. Local corruption simulation with non-destructive recovery.

### Adversarial/Property/Fuzz (Mandatory)

1. Malformed/forged event envelope rejection.
2. Replay/out-of-order message handling.
3. Storage corruption and interrupted-write recovery.
4. Relay flap and duplicate subscription stress.

### Manual Security Acceptance

1. Private key reveal/export works deterministically with clear policy enforcement.
2. No infinite loading states in identity/publish/add/request core flows.
3. Clear degraded/offline messaging for users.
4. Two-user daily flow can be completed reliably by non-technical users.

## Go/No-Go Criteria Before Community Expansion

1. Core security test pack green and reproducible.
2. No high-severity identity/session/key-management regressions.
3. Outbox, sync, and storage reliability metrics non-regressive versus baseline.
4. Security runbook and incident procedures updated and validated.

## Non-Goals in This Cycle

1. No financial or blockchain transaction features.
2. No super-app expansion.
3. No broad social ranking/recommendation overhauls before security gates.

## Dependencies for Later Community System Work

Community milestones can start only after:

1. Identity/device/session contracts are stable.
2. Group E2EE baseline and membership security model are validated.
3. Relay + storage reliability pack passes release gates consistently.
