# 24 v1.0.6 CP3 Soak Matrix

_Last reviewed: 2026-03-23 (baseline commit 17eacc9)._

This matrix defines the manual long-session and two-device replay evidence required before `v1.0.6` release preflight/tagging (`CP3` in `docs/23-versioned-phase-plan-v1.0.4-v1.0.6.md`).

## Scope

1. Long-history in-chat search jump stability.
2. Route liveness while chat navigation and history hydration are active.
3. Search-jump diagnostics convergence under repeated load.

## Session Setup

1. Use one desktop runtime + one web runtime with the same test account.
2. Ensure at least one conversation has deep history (load-more required).
3. Keep console available for copy commands.

## Replay Steps

1. Clear app-event buffer:
: `window.obscurAppEvents?.clear()`
2. Navigate repeatedly for 10-15 minutes:
: `Chats -> Network -> Groups -> Settings -> Chats`.
3. In a long-history chat, run at least 15 search-jump actions:
: include normal id hits,
: include older-result jumps that require `Load More`,
: include voice-note result jumps when available.
4. During replay, force pagination and UI churn:
: scroll to top, click `Load More`, reopen search panel, and jump again.
5. After replay, export one-copy stabilization bundle:
: `copy(window.obscurM4Stabilization?.captureJson(400))`

## Pass Criteria

1. No unrecoverable blank-page or frozen-navigation state.
2. Search-jump actions visibly move viewport to target region.
3. `searchJump.summary.riskLevel` is not `high`.
4. If unresolved events exist, reason codes are actionable (`target_not_found_after_load_attempts` etc.) and reproducible.

## Escalation Criteria

Escalate as active blocker if either occurs:

1. `searchJump.summary.domUnresolvedCount > 0`.
2. Route becomes unresponsive during replay.

When escalating, attach:

1. `copy(window.obscurM4Stabilization?.captureJson(400))`
2. `copy(window.obscurM0Triage?.captureJson(300))`
3. short repro timeline (timestamped user actions).
