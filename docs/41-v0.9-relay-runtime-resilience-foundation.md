# v0.9 Relay Runtime Resilience Foundation

_Last reviewed: 2026-03-15 (baseline commit ab08104)._

## Purpose

This document defines the foundation work needed to make relay connectivity, request delivery, and multi-window runtime behavior stable enough for real use.

It exists because the current branch still treats relay failure mostly as a UI symptom instead of a first-class runtime domain.

The goal is not "retry harder."

The goal is a desktop-first runtime that:

- remains usable when individual relays flap,
- converges on delivery state without page refresh,
- preserves one-window/one-profile isolation,
- degrades honestly when network evidence is missing,
- and can recover without forcing the user to reload the app.

The branch is moving toward v0.9.0 beta readiness, but relay fault tolerance remains the final hard reliability domain before feature expansion.

## v0.9 Relay Stability Solution (Execution Plan)

### Objective

Ship a convergence-first relay resilience slice that:

- keeps one runtime owner per window,
- prevents churn amplification under partial outage,
- exposes evidence-based transport truth to runtime/UI,
- and converges delivery without manual refresh.

### Phase 1: Runtime Truth Surfaces (first)

1. Add a relay transport journal service under relays runtime scope.
   - Track desired subscriptions, last replay attempt/result, and reason codes.
   - Track outbound pending counters from queue owners (DM queue + request outbox) and feed runtime snapshot.
2. Wire journal into `relay-runtime-supervisor`.
   - Replace placeholder `pendingOutboundCount=0` with evidence-backed counts.
   - Publish `lastSubscriptionReplayAt`, `lastSubscriptionReplayResult`.

## Messaging Transport Stabilization Update (2026-03-13)

The branch now enforces a **single canonical incoming transport owner per window** and deterministic degraded-send behavior under relay instability.

Landed model and behavior:

1. Runtime singleton owner for incoming transport.
   - `UnlockedAppRuntimeShell` now mounts a runtime-level messaging transport owner provider once per window.
   - That owner is the only controller instance allowed to run incoming DM subscription/sync and automatic queue processing.
   - Owner activation is gated to unlocked identity plus runtime phase `activating_runtime | ready | degraded`.

2. Non-owner controllers are send-only.
   - Main auxiliary surfaces (global dialogs, search, network, invite/request cards) now run with incoming transport and auto queue processing disabled.
   - Route-dependent ownership toggles were removed from dialog-level components.

3. Strict durable send semantics with deterministic preflight queueing.
   - Durable minimum remains: `>=3 target relays => require 2 successes`, otherwise require `1`.
   - If currently writable scoped relays cannot satisfy that minimum, send is queued immediately with typed reason `insufficient_writable_relays`.
   - The runtime never reports final success without durable evidence.

4. Owner-scoped diagnostics and runtime invariants.
   - Incoming seen/processed, sync start/complete, and queue processor logs now include `transportOwnerId` and `controllerInstanceId`.
   - Runtime diagnostics track active incoming owners and active queue processors per window.
   - Invariant event emits warn severity when incoming owner count is not exactly `1`.
   - DevPanel now exposes these counts for triage.

5. Behavior under third-party relay degradation.
   - External outages (`503/520`, cooldown windows, not-connected send paths) are treated as environmental transport degradation.
   - Local behavior is deterministic: queue early when durable scope is not currently writable, keep retry ownership in the runtime singleton, converge when healthy relay capacity returns.
   - This improves recoverability and removes duplicate owner races, but does not guarantee third-party relay uptime.

## Current Diagnosis

Observed during v0.9 beta recovery work:

- relay connectivity can oscillate between `connected`, `connecting`, and `0 writable relays`,
- the same desktop window may need manual refresh to regain a useful relay state,
- sender UI can show relay-backed publish success while recipient convergence still fails,
- desktop window B can receive some traffic while still missing traffic from window A,
- relay instability and multi-window behavior compound each other.

This means the current system still has three structural weaknesses:

1. relay lifecycle is not owned by one explicit per-window runtime authority,
2. send-time truth and receive-time truth are not modeled as one convergence system,
3. recovery is still too implicit, too scattered, and too dependent on user refresh.

## Audit Note (2026-03-12)

Context-recovery audit of the live relay stack found that the branch now has a clearer relay runtime direction, but the implementation still falls short of the intended single-owner resilience model.

Recovered architecture chain:

- relay configuration is profile-scoped in `apps/pwa/app/features/relays/hooks/use-relay-list.ts`,
- live sockets/publish/subscription behavior is owned by `apps/pwa/app/features/relays/hooks/enhanced-relay-pool.ts`,
- readiness/recovery state is summarized by `apps/pwa/app/features/relays/services/relay-recovery-policy.ts`,
- the per-window runtime snapshot is projected by `apps/pwa/app/features/relays/services/relay-runtime-supervisor.ts`,
- desktop/native transport may bypass browser WebSockets through `apps/pwa/app/features/relays/hooks/native-relay.ts` and `apps/desktop/src-tauri/src/relay.rs`.

Key findings from this audit:

1. The runtime supervisor is still mostly observational, not fully authoritative.
   - It reflects pool/recovery state, but it does not own a subscription journal.
   - `pendingOutboundCount` is still hard-coded to `0`, so the runtime snapshot does not actually represent queued transport truth.
2. Relay health/UI can overstate real communication health.
   - Settings currently derive "healthy/degraded" mainly from open enabled sockets and latency.
   - This can report healthy relay connectivity while recipient convergence is still broken.
3. The stale-subscription watchdog can be fooled by generic inbound traffic.
   - `lastInboundEventAtUnixMs` advances on any inbound relay string message, including `OK` responses and other non-message traffic.
   - Recovery therefore tracks transport chatter, not necessarily useful recipient-side event freshness.
4. Relay scoring currently contains a concrete normalization bug.
   - Relay health metrics store `successRate` as `0..100`, but relay-selection scoring clamps it as if it were `0..1`.
   - This compresses most non-zero relays to the same score band and can distort relay ordering/failover decisions.
5. Transient relay ingestion is still too trusting.
   - `nprofile` hints, NIP-65 relay lists, and legacy kind-3 relay lists can flow directly into transient relay connection attempts.
   - These paths currently lack a strict shared relay-URL trust gate before connection attempts.
6. Native desktop relay commands accept broad relay targets.
   - The desktop relay commands can DNS probe / TCP connect / WebSocket connect arbitrary parsed URLs.
   - Combined with unvalidated relay-hint ingestion, this creates a privacy and SSRF-style risk surface that should be narrowed.
7. Relay-list discovery trust is weaker than DM event trust.
   - Incoming DM handling verifies signatures before routing.
   - NIP-65 / relay-list ingestion paths do not currently apply the same signature-verification discipline before caching relay hints.
8. The relay gateway remains a thin bridge, not a resilience layer.
   - It forwards to one upstream relay, has an unbounded pending queue before upstream open, and has no real upstream recovery/backpressure strategy.

Recommended next recovery order from this audit:

1. Fix relay selection/scoring correctness before further relay-policy tuning.
2. Add one strict shared relay URL validation/trust gate for configured, discovered, and transient relays.
3. Require signature verification before accepting/caching relay-hint events.
4. Upgrade watchdog freshness to distinguish generic inbound chatter from actual subscription/event progress.
5. Move subscription desired-state and queued outbound truth under the runtime supervisor boundary instead of leaving them implied by lower layers.

## Focused Hardening Slice Landed (2026-03-12)

The current branch now includes a narrow relay hardening slice without changing the overall relay architecture.

Landed in this slice:

1. Verified NIP-65 relay-hint ingestion.
   - Incoming kind `10002` relay-list events now require valid event structure, normalized identity, valid signature, and at least one trusted `wss://` relay before cache mutation.
   - Invalid or unsigned relay-list events no longer overwrite previously valid cache state.
2. Shared trust gate for discovered relay hints.
   - Untrusted relay-hint sources now converge on the same strict `wss://` validation path before transient connection or cache updates.
3. Message-vs-event freshness split.
   - Relay runtime now tracks generic inbound message activity separately from inbound routable event activity.
   - Stale-subscription recovery is keyed to useful event freshness, not generic control chatter alone.
4. Runtime evidence propagation.
   - Recovery/runtime snapshots now expose the additional freshness evidence so higher runtime layers can reason about "relay alive" vs "useful events still flowing."

This does **not** complete the full resilience foundation.

Still not solved by this slice:

- subscription journal ownership,
- real outbound queue truth in the runtime snapshot,
- full desktop multi-window transport proof,
- and broader relay/runtime convergence across all request/message paths.

## One-Way DM Investigation Note (2026-03-12)

Latest two-window desktop repro showed a narrower asymmetry after native-session wire compatibility was repaired:

- identity/session mismatch spam was resolved,
- incoming modern DM events were visible on the receiver,
- but reverse-direction real-time delivery could still fail when recipient relay discovery repeatedly returned zero scope.

Root-cause direction from this repro:

- outgoing send scope had no recipient evidence source beyond NIP-65/kind-3 discovery + recipient write relays,
- when those were empty, publish degraded to sender-only relay scope,
- sender-only scope can be asymmetric across windows/profiles and can miss relays where the peer is actually observable.

Landed mitigation for this failure mode:

- receive paths now preserve inbound relay URL evidence per peer (profile-scoped, trusted `wss://` only),
- outgoing relay targeting now includes inbound peer relay evidence when discovery scope is empty,
- diagnostics now carry inbound relay URL and relay scope source so asymmetric routing can be triaged from runtime evidence.

Additional operator usability hardening landed in the same recovery window:

- app-event logging now applies bounded per-event rate limits so repeated transport/decrypt warnings do not overwhelm DevTools or hide actionable signals,
- high-frequency native crypto fallback failures now route through log-hygiene classification instead of raw repeated `console.error`,
- and repeated `next/image` fill-parent warnings in active messaging onboarding/empty-state paths were removed by fixing missing `relative` wrapper positioning.

## Relay Churn Investigation Update (2026-03-13)

Two-window desktop repro remained blocked after the one-way targeting mitigation because relay transport was still unstable under failure churn:

- repeated `connected -> closed -> reconnect` loops per relay URL,
- repeated long-latency `connected` logs from stale in-flight sockets,
- repeated native relay hard failures (`Tor proxy connect failed`, `403`, Cloudflare challenge),
- and callback-id/noise amplification during reconnect storms.

Landed transport stabilization slice:

1. Active-socket generation gating in the relay pool.
   - Each relay connection attempt now has a per-URL generation token.
   - Stale socket `open/error/close/message` events are ignored and no longer mutate live relay state.
2. Reconnect cadence hardening.
   - Added in-flight reconnect dedupe and minimum reconnect interval suppression.
   - Relay reconnect attempts now avoid rapid parallel retries for the same URL.
3. Hard-failure cooldown path.
   - Native hard failure signatures (Tor proxy / Cloudflare challenge class) now enter a temporary relay cooldown window instead of immediate retry hammering.
4. Native relay lifecycle hygiene.
   - Native close/disconnect now avoids duplicate disconnect calls when not connected.
   - Native connect failures emit structured error detail for downstream classification and cooldown gating.

Remaining known risk:

- this does not guarantee relay reachability when upstream relays are externally blocked; it prevents local runtime churn from magnifying that outage and preserves healthier relays as transport candidates.

## Reconnect Ownership Correction (2026-03-13)

Further recovery work removed one core source of relay/runtime split-brain:

- native desktop relay transport no longer runs its own background reconnect scheduler,
- reconnect authority now stays in the JS relay runtime (`enhanced-relay-pool`) as the canonical per-window owner,
- native relay runtime remains a typed connect/disconnect/send/subscribe boundary with persistent subscription state, but without autonomous reconnect loops.

Why this matters:

- dual reconnect owners (JS + Rust) can race each other and repeatedly flap a relay between connected/closed states,
- those races can surface as `Not connected` send failures, delayed delivery convergence, and noisy callback-id churn during live desktop sessions,
- removing the extra reconnect owner reduces lifecycle contention and keeps transport recovery diagnosable from one canonical path.

This correction does not solve upstream relay outages; it removes local reconnect competition so runtime behavior is more deterministic under failure.

## Native Connect Timeout Contract (2026-03-13)

Follow-up hardening aligned native relay connect behavior with the frontend command timeout contract:

- `connect_relay` is now explicitly time-budgeted in native runtime (Tor and non-Tor paths),
- Tor connect attempts run within a bounded command budget instead of open-ended retry loops that can outlive frontend invoke timeouts,
- timeout failures now return deterministic native errors within the same command window, so reconnect policy can react from one completed outcome.

Why this matters:

- previously, frontend could time out `connect_relay` while Rust was still running connect attempts, creating stale completion races and relay flap churn,
- bounded command completion preserves one owner and one outcome per connect attempt, which is required for stable reconnect behavior under load.

## Native Keepalive and Open-Dedupe Hardening (2026-03-13)

Additional transport hardening addressed two runtime-level churn amplifiers:

- native relay runtime now explicitly handles websocket control frames (`Ping` -> `Pong`) in the Rust relay read/write path,
- native relay client open-notification handling now dedupes overlapping `connected` status + `Already connected` command outcomes in one lifecycle window.

Why this matters:

- without explicit pong handling, long-lived native relay sessions can be dropped by relay heartbeat expectations, causing artificial close/reconnect loops,
- duplicate open events can cause redundant lifecycle transitions, extra subscription replay, and misleading duplicate connect telemetry under reconnect pressure.
- stale-generation cleanup in JS relay pool can otherwise call `close()` on outdated native relay wrappers and unintentionally disconnect the currently active URL-scoped native transport.

Additional guard now landed:

- stale relay handles are disposed non-destructively when a replacement socket already owns that relay URL, preserving the active native connection during generation rollover.

## Messaging Render-Loop Guard (2026-03-13)

Desktop runtime triage surfaced a separate app-layer failure that amplified transport instability:

- `Maximum update depth exceeded` in `NostrMessengerContent` after repeated relay churn,
- stack rooted in `main-shell.tsx` connection-derivation effect and `messaging-provider.tsx` state wrapper setters.

Landed guardrails:

2. Deterministic DM-connection derivation.

Why this matters:

- this crash path could force repeated re-render/mutation cycles during relay churn, obscuring real transport failures and contributing to `send_failed_not_connected` noise.
- removing the loop restores one canonical update path for derived DM connection state.

## Tor Degradation Noise And Transport State Guard (2026-03-13)

Follow-up desktop runtime hardening addressed Tor-related failure storms that still produced excessive warnings after the render-loop fix:

- repeated native relay `Not connected` sends against stale-open wrapper state,
- reconnect churn on repeated 503/status-timeout style relay failures,
- and high-severity session-mismatch receive logs overwhelming dev diagnostics.

Landed in this slice:

1. Native relay error-state closure alignment.
   - Native relay `status=error` now deterministically transitions the wrapper into closed state.
   - This prevents upper layers from continuing write attempts on dead native transport handles.
2. Tor/hard-failure cooldown expansion.
   - Hard-failure matching now includes `HTTP 503`, relay status error, and connect-timeout signatures used in current Tor-path outages.
   - Affected relays enter manual cooldown instead of immediate reconnect thrash.
3. Decrypt-noise classification refinement.
   - Known mixed-wire/malformed payload signatures (`Version not found`, `unknown version`, base64/json decode failures) are now classified as expected malformed traffic rather than actionable regression by default.
4. Native session mismatch receive severity/throttling.
   - `identity_mismatch_receive` is now warn-level and transition-throttled instead of per-event error-level spam.

Why this matters:

- Tor/unhealthy-relay outages remain visible, but runtime behavior is now less likely to collapse into warning storms that obscure true transport ownership defects.

## Native Send-Timeout Closure Guard (2026-03-13)

Additional follow-up hardening addressed a remaining relay failure loop seen under Tor-path instability:

- native `send_relay_message` timeout failures could leave relay wrappers in an effectively-open state,
- that allowed repeated write attempts on unhealthy relay handles, producing sustained `native_relay.send_failed` suppression storms.

Landed adjustment:

1. Native send timeout handling now follows transport-failure closure path.
   - Send timeout/not-connected failures are treated as degraded transport failures.
   - Relay wrapper now emits close on send failure so pool-level reconnect/cooldown policy can take over.
2. Send-failure logging policy tightened for degraded transport failures.
   - Timeout/not-connected send failures now use stricter rate-limiting windows to reduce console flood while preserving diagnostics.
3. Incoming decrypt failure severity is now context-aware.
   - Unknown/non-accepted senders with decrypt failures no longer default to regression severity; they are treated as expected malformed/foreign traffic unless trust context indicates otherwise.

Why this matters:

- keeps relay write behavior aligned with actual transport health and prevents stale-open send loops during prolonged upstream outage windows.

## Native Relay Write-Queue Backpressure Guard (2026-03-13)

Follow-up runtime triage identified another transport instability source:

- native relay commands (`send_relay_message`, `subscribe_relay`, `unsubscribe_relay`, `publish_event`) could block on a saturated per-relay write channel,
- frontend invoke timeouts then fired first, causing timeout churn and callback-id noise while native work was still pending.

Landed fix:

1. Native relay command enqueue is now fail-fast.
   - Command paths now use non-blocking enqueue semantics and return deterministic `Relay send queue saturated` / `Not connected` outcomes instead of waiting indefinitely.
2. Native relay write loop is now time-bounded.
   - Relay write task applies a bounded send timeout; prolonged websocket write stalls force connection teardown so reconnect policy can recover cleanly.
3. JS transport classification now treats queue saturation as degraded transport.
   - Queue-saturation errors are handled like transient not-connected conditions for reconnect/write-block behavior.

Why this matters:

- prevents command-timeout stalls from cascading into prolonged send dead-zones and reduces callback-id churn during relay pressure events.

## Inbound Request-Inbox Admission and Replay Dedupe (2026-03-13)

Two-window runtime repro surfaced a critical inbound-path failure under relay churn:

- unknown/unaccepted traffic from public relays was repeatedly entering request-inbox flow,
- some events were routed as unknown-request traffic and then still treated as accepted conversation messages,
- reconnect replay delivered duplicate copies of the same logical message event,
- and repeated request/chat state mutation amplified UI processing cost until window instability/crash.

Landed containment in the canonical inbound DM handler:

1. Unknown non-request DMs are now dropped.
   - Unaccepted peers without request evidence or connection-control tags no longer mutate requests inbox, message persistence, or conversation UI.
2. Request/context events return early through request-only routing.
   - Connection request/accept/decline/cancel/receipt and pending-outgoing context are handled in request paths and do not fall through into accepted-chat mutation.
3. Per-window handled-event dedupe is now explicit.
   - Incoming processing keeps a bounded handled ID set for both wrapper event IDs and decrypted rumor IDs.
   - Duplicate replay from reconnect storms or multi-relay overlap now exits early before request inbox churn and duplicate UI work.

Validation coverage added:

- unknown non-request unaccepted DM drop behavior,
- duplicate connection-request replay suppression,
- gift-wrap rumor replay suppression across distinct wrapper event IDs.

## Non-Goals

This foundation track does not try to solve everything at once.

It does not require:

- perfect relay uptime,
- a custom relay server rollout before client recovery is fixed,
- immediate web/mobile parity,
- or replacing Nostr with a different transport.

It does require the desktop runtime to become reasoned about as a resilient transport client instead of a React app that happens to open WebSockets.

## Architectural Principles

### 1. One runtime owner per window

Each window must have one canonical runtime supervisor for:

- relay startup,
- relay recovery,
- subscription replay,
- sync/backfill scheduling,
- queued send draining,
- and degradation state.

Relay lifecycle must not be split across:

- provider mount effects,
- controller-local retries,
- invitation dialogs,
- sync hooks,
- and ad hoc refresh logic.

### 2. Relay health is transport state, not UI decoration

The relay layer must own:

- connection state,
- writable/subscribable capacity,
- publish acknowledgements,
- inbound traffic freshness,
- circuit-breaker state,
- and recovery stage.

UI components may read this state, but they should not decide transport policy.

### 3. Delivery claims must follow evidence

The system must distinguish:

- `queued_locally`,
- `published_to_some_relays`,
- `published_with_required_scope`,
- `seen_by_recipient_side`,
- `accepted`,
- `conversation_active`.

This is especially important for requests/invitations.

Sender-side relay `OK` is useful evidence, but it is not recipient convergence.

### 4. Recovery must be sticky

When relay connectivity drops, the system should:

- keep local queued work,
- keep subscriptions as desired state,
- aggressively but safely recover,
- and replay the intended runtime state after recovery.

The user should not need to refresh the page to restore subscriptions or drain the queue.

### 5. Relay churn must not poison the runtime

Closing a socket, losing one relay, or hitting one relay's `503` must not:

- tear down the whole messaging runtime,
- advance sync checkpoints incorrectly,
- or reset user-visible flows into dead states.

## Required Foundation Model

### A. Window Relay Supervisor

Introduce an explicit per-window `RelayRuntimeSupervisor`.

Minimum responsibilities:

- own desired relay set for the bound profile,
- own actual relay connection set,
- own the subscription journal,
- own queued publish drain scheduling,
- own recovery stage transitions,
- publish one immutable runtime snapshot to React.

Suggested states:

- `booting`
- `connecting`
- `healthy`
- `degraded`
- `recovering`
- `offline`
- `fatal`

Suggested recovery stages:

- `reconnect_relays`
- `replay_subscriptions`
- `repair_sync`
- `drain_outbox`
- `subsystem_recycle`

### B. Subscription Journal

All live subscriptions should be recorded as desired state, not only as active sockets.

The journal should include:

- subscription owner,
- filter,
- reason,
- creation time,
- replay policy,
- and whether replay is required after reconnect.

On reconnect or subsystem recycle:

- replay subscriptions from the journal,
- do not depend on component remounts,
- and do not require manual navigation to resubscribe.

### C. Sticky Send Queue

Requests and DMs should pass through one queue contract:

- local persistence first,
- publish when writable relays exist,
- bounded retry with backoff,
- surface actual evidence state,
- never lose queued work on temporary disconnect.

This queue should be profile-scoped and window-safe.

The UI should present:

- `queued`,
- `sending`,
- `partial`,
- `published`,
- `needs_attention`,

instead of toggling between "blocked" and "try again manually."

### D. Sync and Backfill as a Service

Backfill should not be a side effect of whichever component happens to be open.

The sync service should own:

- timeline checkpoints,
- replay windows,
- stale inbound detection,
- backfill scope,
- and repair policy.

Checkpoint policy must remain conservative:

- never advance on timeout,
- never advance to wall clock without observed evidence,
- always preserve enough replay window for delayed relay propagation.

### E. Publish Scope Contract

Before a request or DM is marked `published`, the runtime should know:

- which relay URLs were targeted,
- which relays accepted,
- whether recipient-discovered relays were included,
- and whether the publish scope met the current policy.

This means "delivered" should eventually be split into:

- `relay_published`,
- `recipient_scope_published`,
- `recipient_evidence_seen`.

The current branch only partially models this.

### F. Relay Set Strategy

The runtime should separate relay roles:

- baseline stable relays,
- profile-configured relays,
- recipient-discovered relays,
- temporary fallback relays.

Each role should have different retention and recovery rules.

Suggested policy:

- keep a small stable baseline always eligible,
- add recipient relays opportunistically for reach,
- keep fallback relays transient,
- evict unhealthy transient relays more aggressively than configured relays.

## Recovery Policy Design

The recovery system should become explicit and monotonic.

### Trigger sources

- socket close/error,
- zero writable relays,
- stale inbound window,
- queued publish starvation,
- app returns online,
- window gains focus,
- visibility returns,
- explicit user "repair now" action.

### Recovery sequence

1. reconnect missing relays,
2. replay desired subscriptions,
3. run bounded sync repair/backfill,
4. drain queued requests and DMs,
5. only then consider subsystem recycle,
6. only then surface reload as last resort.

### Recovery output

The runtime snapshot should expose:

- `readiness`,
- current recovery stage,
- attempt count,
- last successful inbound time,
- last successful publish time,
- queue backlog,
- replayed subscription count,
- and last reason for degradation.

## Desktop Multi-Window Requirements

Desktop multi-profile makes the transport problem harder because multiple windows share one app process.

The runtime foundation must guarantee:

- no shared implicit active profile,
- no shared message queue ownership,
- no shared subscription ownership,
- no shared native session ambiguity,
- and no transport singleton assumptions that couple windows together.

Allowed:

- shared code,
- shared typed contracts,
- shared native command surface.

Not allowed:

- shared ambient mutable relay runtime state that makes one window's transport lifecycle affect another window's correctness.

## Server/Relay Considerations

Some instability is external.

Example from current logs:

- `wss://relay.damus.io` returns `503`,
- specific relays flap,
- latency spikes under multiple active windows.

The client foundation must assume relays are unreliable.

That means:

- quorum/publish policy must tolerate partial relay acceptance,
- sync policy must tolerate delayed propagation,
- health scoring must de-prioritize unstable relays,
- baseline transport should work even when one default relay is degraded.

Longer-term, the product may need:

- a curated default relay pool with stronger uptime guarantees,
- first-party relay health telemetry,
- optional first-party relay participation,
- and explicit relay tiering by reliability class.

But client runtime recovery must be fixed before server-side improvements can be trusted as the primary answer.

## Immediate Implementation Priorities

### Priority 1: Relay runtime ownership

Move recovery/reconnect/subscription replay logic under one supervisor service per window.

Exit criteria:

- relay reconnect does not depend on page refresh,
- subscriptions replay automatically,
- queue draining resumes automatically.

### Priority 2: Sticky send truth

Route invitation/request/DM sends through one queued transport contract that survives temporary relay outages.

Exit criteria:

- no hard UI dead-end when writable relays drop to zero,
- queued sends recover automatically once writable relays return.

### Priority 3: Recipient convergence diagnostics

Extend diagnostics so maintainer logs can answer:

- what event was published,
- to which relays,
- whether recipient-scoped relays were included,
- whether the recipient window saw the event,
- whether it was ignored, undecryptable, or routed.

Exit criteria:

- no more relying on generic relay logs to guess what happened.

### Priority 4: Relay-set policy hardening

Separate baseline/configured/recipient/fallback relay roles and tighten eviction/recovery rules.

Exit criteria:

- one unstable relay does not drag the whole window into a semi-dead state.

## Acceptance Bar

This foundation work is only successful when:

1. both desktop windows can remain open for extended testing without manual refresh,
2. relay loss causes visible degradation and automatic recovery, not dead UI,
3. queued requests/messages resume automatically after relay recovery,
4. sender and receiver convergence can be diagnosed from explicit runtime evidence,
5. one relay returning `503` does not collapse the entire communication path,
6. desktop multi-window transport behavior remains profile-isolated and deterministic.

## Maintainer Guidance

Do not keep solving this as a sequence of isolated bugs.

Treat relay/runtime resilience as core infrastructure with explicit ownership, typed state, and staged recovery.

If a future fix cannot answer:

- who owns relay lifecycle,
- where desired subscription state lives,
- what evidence is required for delivery claims,
- and how recovery resumes without refresh,

then it is probably another local patch instead of foundation repair.
