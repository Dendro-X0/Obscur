# 18 v0.9.3 Execution Plan

_Last reviewed: 2026-03-21 (baseline commit 68cb62f)._

This document is the concrete execution plan for `v0.9.3`.

## Release Thesis

- `v0.9.3` is a reliability and trust release.
- Goal: make existing messaging and community flows consistently usable before adding high-risk new systems.
- Priority order:
1. Stability and startup/navigation liveness.
2. Cross-device history and media integrity.
3. E2EE trust posture hardening and clearer security UX.
4. UX and performance polish for existing features.

## Non-Goals for v0.9.3

- No full real-time voice production rollout.
- No full community architecture rewrite.
- No additional startup owner chain.
- No parallel sync pipeline added beside current canonical owners.

If voice work begins in this cycle, it must be a bounded research spike with no user-facing production dependency.

## Milestone Plan

## M0 - Baseline Lock and Triage Instrumentation

Objective:
- Freeze execution scope and make regressions diagnosable before behavior changes.

Scope:
- Confirm canonical owner boundaries:
1. startup chain: `DesktopProfileBootstrap -> AuthGateway -> ProfileBoundAuthShell -> UnlockedAppRuntimeShell -> RuntimeActivationManager`.
2. relay/runtime owner: `relay-runtime-supervisor` and `enhanced-relay-pool`.
3. account sync owner: encrypted backup service + projection runtime contracts.
- Add any missing diagnostics needed to classify:
1. startup stalls,
2. route freeze,
3. restore drift,
4. media hydration mismatch.

Acceptance:
1. `pnpm release:test-pack -- --skip-preflight` passes.
2. `pnpm docs:check` passes.
3. One reproducible triage script exists for runtime capture:
   - `window.obscurWindowRuntime?.getSnapshot?.()`
   - `window.obscurRelayRuntime?.getSnapshot?.()`
   - `window.obscurRelayTransportJournal?.getSnapshot?.()`
   - `window.obscurAppEvents?.getDigest?.(300)`
   - preferred one-shot export: `copy(window.obscurM0Triage?.captureJson(300))`

## M1 - Stability and Session Continuity

Objective:
- Eliminate restart/login and page transition failure classes that block basic usage.

Scope:
- Stabilize remember-me/session continuity for both credential and private-key login paths.
- Keep startup fail-open deterministic on profile/key mismatch and blocked storage cases.
- Guarantee route changes remain interactive during relay degradation.
- Prevent blank-page and infinite-loading loops under startup and navigation churn.

Primary modules:
- `apps/pwa/app/features/auth/*`
- `apps/pwa/app/features/runtime/components/profile-bound-auth-shell.tsx`
- `apps/pwa/app/features/runtime/components/runtime-activation-manager.tsx`
- `apps/pwa/app/features/main-shell/*`

Acceptance:
1. 20 restart cycles on desktop and web with remembered session:
   - no forced re-login unless identity mismatch is explicit.
2. 50 route transitions under mixed relay health:
   - no frozen navigation,
   - no unrecoverable blank page.
3. Existing reliability workflow checks remain green on PR and tag paths.

Current implementation status (2026-03-20):
1. M1 session-continuity hardening is in progress and landed:
   - `AuthGateway` now retries native-session recovery when remember-me is true but no passphrase token candidate exists (`auth.auto_unlock_recovered_native_session`).
   - Desktop title-bar `Lock` now preserves remembered auth tokens when remember-me is enabled for the bound profile.
   - Profile boot stall fallback now always exposes deterministic `Continue to Login` recovery (no forced keep-waiting branch).
2. Automated gates for this slice pass:
   - `pnpm docs:check`
   - `pnpm release:test-pack -- --skip-preflight`
   - `pnpm -C apps/pwa build`
3. Manual M1 acceptance is still required before marking M1 complete:
   - restart-cycle replay,
   - route-transition stress replay under relay degradation.

## M2 - Cross-Device History and Media Integrity

Objective:
- Make sender and receiver history converge reliably across devices, including media.

Scope:
- Harden restore/merge/apply boundaries for self-authored DM continuity.
- Ensure canonical conversation mapping cannot strand outgoing history.
- Close desktop/web parity gap for historical media hydration.
- Preserve community membership visibility consistency after new-device login.

Primary modules:
- `apps/pwa/app/features/account-sync/services/encrypted-account-backup-service.ts`
- `apps/pwa/app/features/account-sync/services/account-projection-runtime.ts`
- `apps/pwa/app/features/messaging/services/message-persistence-service.ts`
- `apps/pwa/app/features/messaging/hooks/use-conversation-messages.ts`
- `apps/pwa/app/features/groups/providers/group-provider.tsx`

Acceptance:
1. Two-device deterministic scenario matrix passes for 10/10 runs:
   - DM outgoing/incoming history,
   - group visibility and membership state,
   - restored media references.
2. No regression in `release:test-pack` account-sync and deterministic DM suites.
3. Manual A/B restore replay shows no self-authored DM drop in targeted conversations.

## M3 - E2EE Trust Hardening and UX Evidence

Objective:
- Improve user-visible security confidence without changing the core cryptographic model.

Scope:
- Make key/profile/session mismatches explicit and actionable.
- Ensure delivery states do not overclaim success without evidence.
- Improve security-relevant UX copy and diagnostics around degraded paths.
- Strengthen trust-related tests around unsupported runtime or degraded publish cases.

Primary modules:
- `apps/pwa/app/features/crypto/*`
- `apps/pwa/app/features/auth/*`
- `apps/pwa/app/features/messaging/controllers/outgoing-dm-publisher.ts`
- `apps/pwa/app/shared/log-app-event.ts`

Acceptance:
1. No optimistic-success claim on evidence-missing send paths.
2. Explicit mismatch UI path exists for identity/session conflicts.
3. E2EE-related deterministic tests pass in both web and desktop-oriented flows.

## M4 - UX and Performance Polish (Existing Features)

Objective:
- Improve responsiveness and usability without introducing new architectural risk.

Scope:
- Chat list and message list performance tuning (virtualization and bounded windows).
- Sidebar/search smoothness and interaction stability.
- Progressive loading polish for heavy conversations and communities.
- Mobile layout refinements and interaction consistency.

Primary modules:
- `apps/pwa/app/features/messaging/components/*`
- `apps/pwa/app/features/messaging/hooks/*`
- `apps/pwa/app/components/*`

Acceptance:
1. No measurable regression in route interaction responsiveness on low-to-mid hardware.
2. Heavy-thread open and scroll operations remain responsive with bounded memory growth.
3. Full `apps/pwa` test suite remains green.

## Optional R&D Spike - Real-Time Voice Feasibility (Non-Blocking)

Objective:
- Decide if voice can be shipped safely in `v0.9.4+`.

Scope:
- Technical spike only:
1. signaling model aligned with current identity and relay constraints,
2. NAT traversal/TURN assumptions,
3. encryption/session model compatibility with current trust contracts,
4. expected operational complexity.

Deliverable:
- One short feasibility note with explicit go/no-go criteria.
- No production code dependency required for `v0.9.3`.

## Release Gates for v0.9.3

Required automated gates:
1. `pnpm docs:check`
2. `pnpm release:integrity-check`
3. `pnpm release:artifact-version-contract-check`
4. `pnpm release:test-pack -- --skip-preflight`
5. `pnpm -C apps/pwa exec vitest run`
6. `pnpm -C apps/pwa build`

Required manual gates:
1. Restart/login continuity (desktop and web).
2. Route transition stress under relay churn.
3. Two-device cross-history and media parity replay.
4. Community membership visibility replay on fresh device.

## Do-Not-Touch Boundaries

These are hard boundaries unless an explicit architecture decision is recorded first:

1. Do not add a second startup lifecycle owner.
2. Do not add parallel account-sync mutation paths.
3. Do not mark success from sender-local optimistic state alone.
4. Do not disable relay/runtime degraded evidence just to suppress UI symptoms.
5. Do not introduce hidden singleton state for profile/account scoping.

## Learning Reference Mapping

Signal patterns to emulate:
1. conservative trust state and explicit security UX.
2. clear delivery semantics based on evidence.

Telegram patterns to emulate:
1. fast perceived performance via local-first rendering.
2. robust pagination and media loading behavior.

Discord patterns to emulate:
1. scalable community list/member navigation ergonomics.
2. stable interaction behavior under large list churn.

## Execution Cadence

Suggested cadence:
1. Week 1: M0 + M1 baseline fixes.
2. Week 2: M2 convergence and media parity.
3. Week 3: M3 trust hardening and M4 UX polish.
4. Week 4: stabilization, regression sweeps, release gating, and tag prep.

At the end of each week:
1. update `ISSUES.md`,
2. update `CHANGELOG.md`,
3. record any scope changes in this document.
