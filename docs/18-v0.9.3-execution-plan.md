# 18 v0.9.3 Execution Plan

_Last reviewed: 2026-03-21 (baseline commit 68cb62f)._

This document is the concrete execution plan for `v0.9.3`.

## Release Thesis

- `v0.9.3` is a reliability and trust release.
- Goal: make existing messaging and community flows consistently usable before adding high-risk new systems.
- Priority order:
1. Stability and startup/navigation liveness.
2. Cross-device history and media integrity.
3. E2EE trust posture hardening and clearer security UX.
4. UX and performance polish for existing features.

## Non-Goals for v0.9.3

- No full real-time voice production rollout.
- No full community architecture rewrite.
- No additional startup owner chain.
- No parallel sync pipeline added beside current canonical owners.

If voice work begins in this cycle, it must be a bounded research spike with no user-facing production dependency.

- Guarantee route changes remain interactive during relay degradation.
- Prevent blank-page and infinite-loading loops under startup and navigation churn.

- `apps/pwa/app/features/runtime/components/profile-bound-auth-shell.tsx`
- `apps/pwa/app/features/runtime/components/runtime-activation-manager.tsx`
- `apps/pwa/app/features/main-shell/*`

   - no forced re-login unless identity mismatch is explicit.
2. 50 route transitions under mixed relay health:

   - no unrecoverable blank page.
3. Existing reliability workflow checks remain green on PR and tag paths.

3. Automated gates for this slice pass:

## M4 - UX and Performance Polish (Existing Features)

Objective:
- Improve responsiveness and usability without introducing new architectural risk.

Scope:
- Chat list and message list performance tuning (virtualization and bounded windows).
- Sidebar/search smoothness and interaction stability.
- Progressive loading polish for heavy conversations and communities.
- Mobile layout refinements and interaction consistency.

Primary modules:
- `apps/pwa/app/features/messaging/components/*`
- `apps/pwa/app/features/messaging/hooks/*`
- `apps/pwa/app/components/*`

Acceptance:
1. No measurable regression in route interaction responsiveness on low-to-mid hardware.
2. Heavy-thread open and scroll operations remain responsive with bounded memory growth.
3. Full `apps/pwa` test suite remains green.

Current implementation status (2026-03-20):
1. M4 sidebar/chat-list render-cost reduction slice landed:
   - `Sidebar` conversation partitioning now uses memoized Set-backed single-pass bucketing for hidden/pinned/direct/community views and unread totals (replacing repeated multi-pass filter/reduce + `includes` scans).
   - `main-shell` chat unread badge aggregation now uses memoized hidden-ID Set filtering and memoized unread reduction.
2. M4 message-row update-isolation slice landed:
   - `MessageList` now passes row-local booleans (`isFlashing`, `isMessageMenuAnchored`, `isReactionPickerAnchored`) into `MemoizedMessageRow` instead of global message-id selectors.
   - This prevents unrelated visible rows from re-rendering when one bubble’s menu/reaction picker opens or when flash target changes.
3. M4 message-list scroll-event pressure slice landed:
   - `MessageList` scroll handler now batches metrics with `requestAnimationFrame` (one scroll-state computation per frame) instead of processing every raw scroll event callback.
   - Added typed scroll-threshold utility contract in `message-list-scroll.ts` with focused tests (`message-list-scroll.test.ts`) for bottom-distance and fast-scroll velocity boundaries.
4. M4 message payload parse-cache slice landed:
   - `MessageList` now parses JSON payloads once per message-list update and reuses cached payloads for both invite-response status derivation and per-row render metadata.
   - Removes duplicate parse passes during high-volume timeline renders.
5. M4 attachment/media derivation slice landed:
   - Added typed helper contracts in `message-attachment-layout.ts` to classify attachment buckets and derive display metadata in a single pass.
   - `MessageAttachmentLayout` now memoizes attachment buckets and presentation maps (display names + hosts), removing repeated filter passes and repeated `new URL(...)` parsing in row render paths.
   - Added focused coverage in `message-attachment-layout.test.ts` for bucket partitioning and display/host fallback order contracts.
6. M4 message render-meta cache consolidation slice landed:
   - Added typed helper contract in `message-list-render-meta.ts` to build parsed payload cache, invite-response mapping, and per-message render metadata in a single traversal.
   - `MessageList` now consumes the unified render cache instead of running three independent full-message loops.
   - Added focused coverage in `message-list-render-meta.test.ts` for payload parsing, invite status mapping, and relay-URL content visibility toggles.
7. M4 message-row time-label re-render isolation slice landed:
   - `MessageList` now computes row `timeLabel` at the visible-list boundary and passes that rendered value to `MemoizedMessageRow`.
   - Row memo equality now compares `timeLabel` instead of raw `nowMs`, preventing unnecessary visible-row re-renders on periodic clock ticks when the displayed label is unchanged.
8. M4 sidebar conversation-row churn reduction slice landed:
   - `ConversationRow` is now memoized with field-level equality for rendered conversation/timestamp/unread inputs.
   - `Sidebar` now computes row-level timestamp labels (`lastMessageLabel`, `lastActiveLabel`, `lastViewedLabel`) before rendering rows and passes stable handler references (`togglePin`, `deleteConversation`) instead of per-row closures.
   - This reduces periodic sidebar row re-renders when the 30s clock tick does not change rendered label text.
9. M4 mobile route-stall hard-fallback parity slice landed:
   - `MobileTabBar` now mirrors desktop sidebar route-guard behavior: on route request, it arms a deterministic hard-navigation fallback and clears it once pathname settles.
   - Route-guard diagnostics now include `guardSource: "mobile_tab_bar"` for triage disambiguation.
   - Added focused coverage in `mobile-tab-bar.test.tsx` for fallback trigger and settle-cancel behavior.
10. Behavior contract remains unchanged (pagination/search/tab semantics preserved) with focused regression signal:
   - `sidebar.test.tsx` remains green.
11. M4 route-settle diagnostics slice landed for freeze triage:
   - `AppShell` now records per-path route mount probe timings (`start`, `slow-threshold`, `settled`) with frame-delay metadata and route-request elapsed latency.
   - navigation diagnostics now include deterministic route-surface tags (`routeSurface`, `fromRouteSurface`, `targetRouteSurface`) for faster page-family attribution during freeze incidents.
   - Added a lightweight runtime diagnostics snapshot surface at `window.obscurRouteMountDiagnostics?.getSnapshot?.()` for incident capture.
   - M0 triage capture now includes route-mount diagnostics snapshot + navigation focused events for route-mount probe anomalies.
   - Added focused coverage in:
     - `app/components/page-transition-recovery.test.ts`,
     - `app/components/app-shell.test.tsx`,
     - `app/shared/m0-triage-capture.test.ts`.
12. Manual M4 acceptance completed on 2026-03-21 (dev-server verification):
   - route interaction responsiveness replay passed on target hardware profile under relay churn stress without unrecoverable navigation lock,
   - heavy-thread/chat scroll responsiveness replay remained interactive with bounded memory behavior in validated test windows.

## Do-Not-Touch Boundaries

These are hard boundaries unless an explicit architecture decision is recorded first:

1. Do not add a second startup lifecycle owner.
2. Do not add parallel account-sync mutation paths.
3. Do not mark success from sender-local optimistic state alone.
4. Do not disable relay/runtime degraded evidence just to suppress UI symptoms.
5. Do not introduce hidden singleton state for profile/account scoping.
