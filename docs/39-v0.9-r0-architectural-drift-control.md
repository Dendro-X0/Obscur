# v0.9 R0 Architectural Drift Control (Contract/Gate Correction)

_Last reviewed: 2026-03-10 (baseline commit ab08104)._
_Current branch-state handoff is recorded in doc 40._

## Purpose

R0 is the stabilization governance slice before deeper transport rewrites.
It does not attempt full feature recovery. It aligns docs, release gates, and deterministic checks with actual runtime behavior.

## Drift Signals (Current)

1. Contract drift: docs/reporting overstate reliability for request delivery.
2. Runtime drift: desktop/web upload and relay behavior diverge under failure.
3. State drift: sender "pending" status may diverge from recipient receipt.
4. Gate drift: release pack lacks several controller-level deterministic tests.

## R0 Scope

1. Correct status claims in recovery docs to match observed behavior.
2. Expand `release:test-pack` with controller tests covering request guards, incoming routing, and subscription churn.
3. Define explicit R0 exit criteria that must pass before marking request delivery as `working` again.
4. Keep community/system expansion blocked while R0/R1 reliability gaps remain open.

## R0 Exit Criteria

1. Docs and changelog reflect current risk profile with no contradictory "working" claims for known flaky flows.
2. Release targeted pack includes:
   - request guard determinism,
   - incoming DM/request routing checks,
   - subscription idempotence/churn checks.
3. Two-user deterministic request flow is tracked as release-blocking for R1 completion (`10/10` consecutive runs).
4. No new release may claim request-delivery recovery without evidence from deterministic two-user runs.

## Current Status (2026-03-10)

R0/R1 work improved contracts, diagnostics, and deterministic failure classification, but the branch is still not through the gate.

Current release blockers still include:

1. sender-local-only invitation/request success can still appear in user-facing desktop flows,
2. multi-window desktop communication remains insufficiently proven end-to-end,
3. multiple overlapping request/invite/runtime paths still exist in the live branch.

R0 should therefore still be considered open.

Read [v0.9.0 Beta Status and Recovery Handoff](./40-v0.9.0-beta-status-and-recovery-handoff.md) for the current maintainer-level summary before making additional release claims.

## Operational Gate

Run before release tagging:

```bash
pnpm release:test-pack
pnpm release:preflight
```

If either fails, release is blocked for reliability claims.

3. Cross-runtime upload/profile URL portability and parity checks.

   - Request acceptance no longer marks receiver state as `accepted` before relay evidence exists; queued/failed accept attempts stay pending and return typed non-success outcomes.

10. Landed DM/profile transport proof tightening for the next hard-gate slice:
   - Outgoing DM send and queued retry no longer claim success when only `sendToOpen` fire-and-forget transport is available; unsupported runtimes now settle to deterministic non-success outcomes instead of optimistic accepted/sent states.

   - Queued DM retry now settles on partial relay evidence instead of re-entering the retry loop, so degraded delivery can converge without duplicate resend storms.
   - Profile network publish now records typed delivery status (`sent_quorum | sent_partial | queued | failed`) in the publish report and no longer marks success through the generic `sendToOpen` fallback.
   - Added hook/publisher regression coverage for partial profile publish, queued no-writable-relay publish, unsupported runtime publish failure, and unsupported DM transport behavior.
   - Added a deterministic two-user DM integration harness (`10/10`) covering: no writable relays -> queued first DM -> sender restart -> partial relay recovery -> receiver restart -> decrypt/receive, with first-divergence reporting.
   - Added regression coverage for native host adapter behavior and background-service fallback behavior.
9. Landed relay-native bridge extraction:
   - Added a dedicated relay native adapter for Tor status, relay command invocation, relay probe, and relay event subscriptions.
