# v0.9.0 Beta Status and Recovery Handoff

_Last reviewed: 2026-03-16 (baseline commit ab08104)._

## Purpose

This document is the maintainer-facing handoff for the unreleased `v0.9.0 beta` line.

It exists to record the actual state of the branch after the v0.9 recovery/foundation work:

1. what has landed,
2. what remains fundamentally unstable,
3. why the branch is not yet releasable,
4. what to investigate first when work resumes.

This is not a release announcement and not a feature-complete beta report. It is a recovery-state snapshot.

## Current Branch Status

`v0.9.0 beta` remains unreleased.

The branch contains substantial foundation work for:

- profile isolation,
- profile-scoped persistence,
- typed runtime/native adapters,
- relay-core publish outcome normalization,
- protocol-core adapter surfaces,
- deterministic discovery foundations,
- request/outbox state normalization,
- delivery diagnostics and sync-checkpoint repair work.

However, the branch still fails the core product bar because end-to-end user-to-user communication is not yet trustworthy under realistic desktop multi-window testing.

## Status Addendum (2026-03-15)

Recent recovery progress improved core communication behavior:

- cross-device chat history synchronization between desktop and web now converges in current test flows,
- prior unread stacking/loop behavior was reduced by projection/account-sync convergence fixes,
- runtime messaging transport ownership generally converges to expected per-window invariants after startup.

Remaining release-critical risk has shifted toward relay fault tolerance, not base message model correctness:

- relay close/reconnect churn can still create degraded runtime phases and intermittent `Not connected` send outcomes,
- relay readiness projection still lacks full outbound truth (`pendingOutboundCount` remains placeholder in relay runtime snapshot),
- scoped publish reliability still depends on global connection readiness checks rather than explicit scoped-connection evidence.

For the active stabilization plan, see:

- [docs/41-v0.9-relay-runtime-resilience-foundation.md](./41-v0.9-relay-runtime-resilience-foundation.md)

## Context Recovery Note (2026-03-12)

This note records a reproduced desktop multi-window failure that was not written down promptly during the previous investigation cycle.

Observed reproduction:

- account A and account B were opened in two isolated desktop windows using the multi-profile runtime,
- account A resolved account B by exact `npub`/public-key search in Discovery,
- account A used the exact-match `Send Invitation` CTA,
- the sender UI reported `Invitation delivered` and retained an outgoing pending request,
- account B did not receive the invitation or follow-up message from account A,
- account B still received unrelated inbound traffic (for example advertisement-style request/inbox items).

Interpretation:

- the recipient window is not fully offline or globally incapable of receiving traffic,
- the branch still contains a narrower convergence failure in request/message delivery between two local test accounts,
- this remains consistent with the existing architectural assessment that sender-local success, recipient convergence, relay scope, and runtime ownership are not yet fully unified.

Code-audit findings recovered during context rebuild:

1. Discovery exact-match send path currently routes through the newer request transport stack:
   - `apps/pwa/app/search/page.tsx` calls `requestTransport.sendRequest(...)` from `sendDirectRequest(...)`.
2. Request inbox/outgoing management UI still renders through the legacy `inviteManager` surface:
   - `apps/pwa/app/components/invites/connection-request-inbox.tsx` and `apps/pwa/app/components/invites/outgoing-connection-requests.tsx` both load via `inviteManager`.
3. The legacy invite manager is bridged back into the newer request transport/state from main-shell wiring:
   - `apps/pwa/app/features/main-shell/main-shell.tsx` configures both `configureInviteRequestTransportBridge(...)` and `configureInviteRequestStateBridge(...)`.
4. Incoming request routing still depends on the DM subscription + decrypt + tag-routing path:
   - `apps/pwa/app/features/messaging/controllers/dm-subscription-manager.ts` subscribes for kinds `4` and `1059` with `#p` filtering,
   - `apps/pwa/app/features/messaging/controllers/incoming-dm-event-handler.ts` decides whether an incoming event becomes a request inbox item, receipt ACK, accept, decline, cancel, or ordinary DM.

Immediate implication:

- this reproduced failure should not be treated as a simple requests UI bug,
- the highest-probability investigation areas remain publish scope to recipient relays, recipient-window subscription ownership/recovery, decrypt/routing outcomes, and overlapping request lifecycle owners.

## What Landed in This Beta Cycle

### 1. Runtime and boundary restructuring

- Runtime capability checks were centralized.
- Native command access moved behind typed adapters in more shared paths.
- Desktop profile/window isolation contracts were added and expanded.
- A per-window runtime supervisor was introduced to reduce bootstrap ambiguity.

### 2. Identity and profile isolation groundwork

- Profile-scoped storage-key derivation was extended across more identity, settings, and request/discovery surfaces.
- Desktop multi-profile support added explicit window/profile binding concepts.
- Native session handling moved closer to profile-scoped ownership instead of one implicit active identity.

### 3. Messaging and request transport hardening

- `SendResult` and request transport outcomes were normalized to deterministic delivery states.
- Request evidence tracking was tightened to distinguish:
  - sender-local pending,
  - recipient-seen pending,
  - accepted/rejected outcomes.
- DM publish and queued retry flows moved onto typed publish outcome mapping rather than pure fire-and-forget assumptions.
- Unsupported evidence-free publish paths were reduced so some former false-success states now fail deterministically.

### 4. Relay and sync reliability work

- Relay-core publish outcome contracts were added.
- Sync checkpoint repair and targeted backfill logic were introduced.
- Queue retry progression and retry floors were tightened to reduce thrash loops.
- Delivery diagnostics were added for per-window subscription/sync/publish/incoming event tracing in dev builds.

### 5. Discovery and exact-resolution groundwork

- Discovery and resolver work shifted toward deterministic exact-token resolution for:
  - pubkey,
  - `npub`,
  - contact-card style inputs.
- Resolver/cache contracts were added to support a cleaner exact-account path.

## Beta Blocking Failures

The following failures still block release.

### 1. Core request/message delivery is not yet trustworthy

Observed branch behavior:

- sender UI can show a locally visible outgoing invitation/request,
- receiver can remain empty,
- sender-local state can imply success without receiver-visible convergence.

Interpretation:

- the current branch still contains conflicting request/message transport paths or conflicting lifecycle owners,
- delivery evidence is not yet sufficient to claim end-to-end success in all active flows,
- transport scope and inbound routing still need deeper consolidation.

### 2. Multiple runtime owners still coexist

The branch still carries overlapping generations of architecture:

- legacy invite/request compatibility bridge,
- newer request transport abstraction,
- legacy/modern DM paths,
- relay-core publish mapping,
- protocol-core adapter path,
- profile/window runtime supervision,
- older provider-driven bootstrap behavior.

This means a single user action may still traverse more than one conceptual system.

### 3. Desktop multi-window remains the hardest instability surface

The desktop-first profile isolation work improved the model, but the branch still shows that:

- second-window behavior can diverge from first-window behavior,
- session/runtime ownership can still be difficult to reason about,
- communication testing in two desktop windows remains the most important unresolved acceptance path.

### 4. Auth/session restore and UI state remain inconsistent in some flows

Known symptoms reported during this beta cycle included:

- repeated re-login requirements,
- stale account chips despite locked/auth-required state,
- restore/profile/account-rehydrate flows surfacing during ordinary navigation,
- onboarding/auth loops after successful import/unlock.

Some of these paths were improved, but the branch should still be treated as unstable until a single runtime owner fully controls auth/session/bootstrap transitions.

### 5. Web runtime remains non-authoritative for release confidence

The desktop app currently carries more of the active testing burden.

The web/PWA runtime has shown:

- hydration/runtime drift,
- indefinite loading states,
- weaker native/runtime parity,
- less reliable confidence for diagnosing desktop-native failures.

`v0.9.0 beta` should not be considered releasable based on desktop-only partial success while web/PWA remains unstable.

## Architectural Assessment

The core problem is no longer well-described as "one bug."

The branch accumulated multiple partially overlapping systems for the same responsibilities:

- auth/session,
- profile isolation,
- runtime startup/teardown,
- request transport,
- legacy invite manager bridges,
- DM publish/receive,
- relay recovery/sync,
- protocol-core rollout behavior.

As a result, the project drifted into a state where local repairs often improved one path while leaving another path alive.

This is the main reason the branch repeatedly produced:

- ghost accounts,
- ghost requests,
- local-only request success,
- restore/auth loops,
- non-deterministic behavior across windows.

## Release Decision

As of 2026-03-10:

- `v0.9.0 beta` is still a recovery branch, not a releasable beta build.
- No release should claim stable decentralized account sync or stable two-user messaging unless the deterministic two-user acceptance path is re-proven in the actual desktop runtime used by testers.

Minimum release claim for this line should require:

1. exact public-key/`npub` resolution works reliably,
2. request send/receive works across two isolated accounts,
3. accept/request state converges on both sides,
4. first-DM send/receive works after connection acceptance,
5. restart does not destroy identity/session state,
6. no false-success local-only request state remains in supported UI flows.

## What To Investigate First When Work Resumes

Do not start with cosmetic UI fixes.

### Priority 1: Collapse request/invitation send paths

Audit every user-facing "send invitation/request" entry point and prove they all route through one canonical transport path.

Specifically investigate:

- discovery page exact-match CTA,
- network profile CTA,
- legacy invite manager bridge,
- request transport service,
- enhanced DM controller connection-request path.

Goal:

- one user action,
- one transport path,
- one source of truth for delivery evidence.

### Priority 2: Prove inbound request routing in the recipient window

Instrument and verify:

- subscription ownership,
- recipient pubkey filter correctness,
- decrypt success/failure,
- request routing to inbox state,
- rejection/ignore paths.

Goal:

- if receiver saw event, docs/logs must show whether it was ignored, undecryptable, or routed.

### Priority 3: Finish runtime ownership consolidation

The per-window runtime supervisor direction is correct, but not all heavy lifecycle paths are fully subordinate to it.

Goal:

- auth/runtime/relay/account-sync startup ordered by one owner,
- no parallel lifecycle inference from multiple providers/hooks.

### Priority 4: Reduce overlapping compatibility layers

Keep compatibility bridges only where they are still necessary.

Remove or quarantine legacy flows that still mutate live request/message state outside the canonical transport path.

## Maintainer Guidance

When returning to this branch later:

1. read docs `37`, `39`, and this doc first,
2. treat the branch as a recovery branch, not a stable feature branch,
3. prefer architectural subtraction and path consolidation over incremental UI repair,
4. do not mark communication fixed unless two-user desktop testing proves both send and receive paths,
5. document every remaining live transport path before changing behavior again.
