# v0.9.x Foundation Recovery Roadmap (Security/Identity Before Community)

_Last reviewed: 2026-03-10 (baseline commit ab08104)._
_See docs 39 and 40 for current recovery-state notes._

## Summary

This roadmap is the active execution baseline for v0.9.x.

1. Stabilize P2P paths first (no infinite loaders, deterministic add/request outcomes).
2. Move security-critical protocol concerns behind Rust-capable adapters.
3. Rebuild identity/E2EE/session/storage contracts with strict rollout flags.
4. Re-enable advanced discovery only after hard security/reliability gates pass.
5. Community expansion is blocked until hard gate criteria are green.

## Locked Decisions

1. Compatibility bridge remains in place during transition.
2. Canonical identity share is signed contact card.
3. Short alias/code is convenience only, not cryptographic source of truth.
4. Rust protocol core is mandatory before full v0.9 feature expansion.
5. Native storage baseline is encrypted SQLite; web-dev uses encrypted adapter bridge.
6. Global search is never required for friend add.
7. Hard security gate is required before community-system work.

## Ticket Waves (Execution Order)

### Wave A (Hotfix Floor, P0)

1. Keep `stabilityModeV090=true` default.
2. Safe Add Friend flow: `contact-card | npub | hex` (plus legacy bridge where available).
3. Split profile operations into independent `Save Local` and `Publish Network` state paths.
4. Add hard action timeouts and terminal states for profile publish + request send.
5. Keep relay health advisory-only; do not block send when identity is resolvable.

### Wave B (Protocol Foundation, P0)

1. Add `protocolCoreRustV090` rollout flag (default off).
2. Add protocol core adapter contract for identity/session/verification/outbox/storage operations.
3. Introduce shared contracts for:
   - identity root/device/session state,
   - handshake + ratchet verification results,
   - quorum publish and storage recovery reports.
4. Keep unsupported runtime behavior typed and non-throwing.

### Wave C (E2EE Rewrite, P0)

1. Add `x3dhRatchetV090` rollout flag (default off).
2. Implement full X3DH + Double Ratchet path behind feature gate.
3. Maintain compatibility bridge for migration window.
4. Add replay/out-of-order reject semantics with deterministic reason codes.

### Wave D (Discovery + Gate Enforcement, P0/P1)

1. Deterministic discovery requires protocol-core gate and stability mode off.
2. Outbox lifecycle remains authoritative:
`queued -> publishing -> sent_partial|sent_quorum -> accepted|rejected|expired|failed`.
3. People/Communities search remains best-effort and non-blocking for friend-add.
4. Optional index remains off-by-default and non-required.

## Hard Gate Before Community Expansion

1. Security test pack green.
2. Reliability chaos pack green.
3. Two-user deterministic flow passes `10/10` consecutive runs.
4. No high-severity identity/session regressions in release candidate.
5. No sender-local-only invitation/request success remains in supported desktop flows.
6. Desktop multi-window request and DM receive paths are proven in the real runtime, not only by unit/hook coverage.

## Implementation Status Snapshot (2026-03-07)

Completed in current branch (still flag-gated where noted):

   - Outgoing send uses protocol-core quorum publish when enabled.

   - Queued sends and immediate sends share one publish contract.

   - Verification is typed (`ok | unsupported | failed`) and non-throwing.
4. Outgoing envelope metadata injection enabled for policy path:
   - `obscur-envelope-version`, `obscur-session-id`, `obscur-counter`.
5. Contact request outbox observability improved:
   - Partial/quorum/failure visibility and bounded transition toasts.
   - Retryable vs non-retryable failure classification persisted in record metadata.
6. Release targeted-pack tooling corrected:
   - `run-release-test-pack` preflight arg forwarding is deterministic.
7. Wave 0 canonical audit matrix published:
   - Added feature/API/runtime failure matrix for relay lifecycle, DM, request, profile, upload, sync, and subscriptions.
8. Relay-core typed publish cutover expanded:
   - DM publish/queued retry and profile publish now route through relay-core typed outcomes before legacy fallbacks.
9. Queue transport determinism pass landed:
   - Offline queue processing now consumes structured send outcomes and no longer loops failed entries without retry progression.
10. Request delivery-state normalization landed:
   - Request outbox now maps deterministic delivery states from send results (`sent_quorum|sent_partial|queued_retrying|failed`).
11. Queue retry schedule hardening landed:
   - Queued DM retry scheduling now enforces a minimum forward delay to prevent immediate repeat attempts under jitter.
12. Upload provider reliability hardening landed:
   - Legacy/stale NIP-96 provider URLs are normalized to current endpoints at runtime.
   - Native upload now retries through direct network path when Tor-proxy transport fails on network-level errors.
13. R0 architectural drift-control kickoff landed:
   - Added R0 control document (`docs/39-v0.9-r0-architectural-drift-control.md`) and synchronized release/testing docs with R0 gate language.
   - Expanded `release:test-pack` with messaging controller determinism tests (request guard, incoming routing, subscription churn).
14. R1 request-delivery reconciliation slice landed:
   - Incoming request control ACK (`connection-received`) path added to refresh sender pending state via recipient evidence.
   - Outbox terminal-failure reconciliation now clears stale outgoing pending guard lockouts.

Still open for completion before hard gate:

1. Finalize full custom X3DH+ratchet engine behavior under `x3dhRatchetV090`.
2. Complete restart-recovery coverage for outbox/session state under churn.
3. Resolve request-delivery state drift (sender pending vs recipient receipt) under relay churn.
4. Finish adversarial/chaos pack and pass `10/10` deterministic two-user run gate.

## Implementation Status Addendum (2026-03-10)

Additional work landed after the 2026-03-07 snapshot:

1. Request-state convergence tightened:
   - sender-local pending without recipient evidence is no longer treated as canonical success,
   - incoming and outgoing pending request surfaces were separated more explicitly.
2. Publish-scope tightening landed:
   - outgoing request/DM flows now carry broader relay scope inputs and no longer rely only on already-open sender relays.
3. Delivery diagnostics landed:
   - per-window diagnostics now expose last subscription, sync, publish, and inbound-event handling in dev builds.
4. DM sync checkpoint handling tightened:
   - sync no longer advances checkpoints on incomplete or timed-out runs.
5. Recipient relay resolution tightened further:
   - recipient relay discovery now returns and caches resolved relay URLs for send-time use.

These changes improved observability and removed some false-success states, but they did not yet close the hard gate for deterministic two-user communication.
