# v0.8.5 Ticketized Roadmap (Auth + Sybil + Ghost Entity Hardening)

_Last reviewed: 2026-03-05 (baseline commit ad788a4)._

_Drafted: 2026-03-05._

This roadmap is a stabilization-first release focused on identity integrity and abuse resistance for one-on-one and community flows.

## 1) Release Theme

1. Make identity handling deterministic and non-ambiguous.
2. Reduce request/join spam and practical Sybil abuse impact.
3. Eliminate local ghost users/ghost communities caused by inconsistent identifiers or malformed events.

## 2) Scope and Guardrails

### In scope

1. Authentication and identity invariant hardening.
2. Contact request and community join anti-spam controls.
3. Canonical identity/community normalization and duplicate suppression.
4. Runtime observability for rejected/quarantined suspicious events.

### Out of scope

1. Full protocol redesign or relay contract changes.
2. KYC/real-world identity features.
3. Perfect Sybil prevention (not feasible in decentralized key-only systems).

## 3) Ticket Backlog

### A. Auth Integrity (P0)

1. `V085-AUTH-001` Canonical Identity Contract Enforcement
   - Enforce normalized `PublicKeyHex` at all identity boundaries (storage, unlock, invite, trust, blocklist, request inbox).
   - Add one shared assertion utility for key-pair consistency checks.
   - Acceptance:
     - no duplicate local identity records from mixed key formats,
     - unlock/import fails fast on key mismatch.

2. `V085-AUTH-002` Session/Device Identity Invariant Audit
   - Audit auto-unlock/native-session paths for stale key drift.
   - Add explicit mismatch handling path (lock + actionable prompt) instead of partial unlock states.
   - Acceptance:
     - no unlocked state when native session key and stored account key differ.

3. `V085-AUTH-003` Identity Repair Tooling (Developer + User Safe Path)
   - Add non-destructive local identity diagnostics and guided repair flow for corrupted identity metadata.
   - Acceptance:
     - maintainers can detect and repair mismatch without wiping unrelated app data.

### B. Sybil and Abuse Resistance (P0)

1. `V085-SYBIL-001` Request Cooldown and Pending-State Enforcement
   - Extend outgoing connection request guards with bounded cooldown windows.
   - Prevent repeated send while pending or immediately after decline/block.
   - Acceptance:
     - repeated click/send attempts do not produce repeated relay events.

2. `V085-SYBIL-002` Community Join Request Abuse Controls
   - Keep join pending lock; add explicit state transitions for accepted/denied/expired.
   - Add per-community cooldown after rejection.
   - Acceptance:
     - user cannot flood same community with repeated join events from same identity state.

3. `V085-SYBIL-003` Optional Risk Signals (No Hard Ban by Default)
   - Add internal risk scoring hints for suspicious patterns (burst requests, repeated malformed identity payloads, rapid identity churn on one device).
   - Keep enforcement policy soft by default (warn/deprioritize/quarantine), not permanent ban.
   - Acceptance:
     - risk signals visible in logs/debug tools with no false hard-lockouts.

### C. Ghost Users and Ghost Communities (P0)

1. `V085-GHOST-001` Canonical Conversation/Community ID Derivation Audit
   - Re-verify all derivation points (`conversationId`, group/community IDs, relay URL normalization).
   - Add shared derivation helpers where drift exists.
   - Acceptance:
     - no duplicate threads/communities from formatting differences.

2. `V085-GHOST-002` Suspicious Event Quarantine Path
   - Quarantine malformed or invariant-violating events (invalid sender, actor mismatch, invalid binding tags) in-memory with bounded logs.
   - Do not create UI entities from quarantined events.
   - Acceptance:
     - malformed events do not create visible ghost users/communities.

3. `V085-GHOST-003` Local State Deduplication Migration
   - Add one-time local state cleanup pass to merge canonical duplicates (requests/trust/block/community references).
   - Keep migration idempotent and reversible via backup snapshot.
   - Acceptance:
     - existing users with dirty state converge to single canonical entries.

### D. Observability and Operations (P1)

1. `V085-OBS-001` Identity/Abuse Metrics
   - Add counters:
     - rejected request attempts,
     - suppressed duplicate join attempts,
     - quarantined malformed events,
     - deduplicated state entries.
   - Acceptance:
     - metrics available in dev tooling/log snapshots.

2. `V085-OPS-001` Incident and Recovery Runbook Updates
   - Document triage for ghost identity/community reports and safe remediation steps.
   - Acceptance:
     - maintainers can resolve user reports with deterministic checklist.

## 4) Suggested Sequencing

1. Wave 1: `AUTH-001/002`, `SYBIL-001`, `GHOST-001` (core invariants first).
2. Wave 2: `SYBIL-002`, `GHOST-002`, `OBS-001`.
3. Wave 3: `AUTH-003`, `GHOST-003`, `OPS-001`.
4. Wave 4: stabilization soak, regression sweep, release notes.

## 5) Test Plan

### Unit

1. Key normalization and key-pair assertion edge cases.
2. Request/join state machine transitions (pending, accepted, declined, blocked, cooldown).
3. Canonical ID derivation and dedupe reducers.

### Integration

1. Repeated add-contact attempts while pending.
2. Repeated join-community attempts while pending/recently rejected.
3. Mixed malformed + valid event streams; verify only valid entities reach UI.
4. Existing dirty local state migration to canonical entries.

### Manual / Profiling

1. Unlock/import with mismatched keys must fail and show actionable path.
2. Long-running session with reconnect cycles should not produce identity drift.
3. Group and DM flows remain smooth with added guards enabled.

## 6) Exit Criteria

1. No duplicate pending requests can be emitted for same peer while unresolved.
2. No duplicate join requests can be emitted for same community while unresolved.
3. Ghost users/communities from key/ID formatting drift are eliminated in tested paths.
4. Mismatched auth state cannot enter unlocked mode.
5. Observability exists for reject/quarantine/dedupe paths.

## 7) Known Limits

1. Decentralized systems cannot fully prevent Sybil identities without external trust anchors.
2. v0.8.5 targets practical abuse resistance and deterministic identity handling, not absolute Sybil elimination.
