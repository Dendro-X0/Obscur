## Work log (newest first)

### 2026-05-15 — **REL-001 — partial: leave intent blocks persisted_fallback resurrection**

- **Symptom:** After leave, restore/login could show a community as joined when backup chat-state still had `createdGroups` but ledger row was missing or stale.
- **Root cause:** `resolveCommunityMembershipRecovery` **persisted_fallback** path materialized groups without ledger coverage and coordinator emitted **`persisted_fallback_backfill` → joined**, bypassing terminal leave when ledger was absent from backup.
- **Fix:** `hasDurableCommunityLeaveIntent` — terminal ledger, tombstone, or **leave outbox** entry suppresses persisted_fallback + backfill. `GroupProvider.leaveGroup` now **enqueues leave outbox** (parity with sealed leave). Vitest: `community-rel-001-leave-intent-guard.test.ts` (3).
- **Remaining:** Stale **joined** ledger row in backup beating **left** by timestamp; relay-side re-join without local leave intent; multi-profile cross-read (REL-003).

### 2026-05-14 — **DM-001 / ARCH-001 — accepted product limitation (investigation closed)**

- **Decision:** Stop incremental debugging on delete-for-me surviving **page refresh**, **dev-server restart**, and **account restore**. The behavior is treated as an **architectural / design foundation** issue, not a single bug.
- **Why patches did not converge:** DM history has **multiple live owners** (account projection event log, IndexedDB, chat-state, SQLite tombstones, encrypted backup restore + convergence). Web and Desktop share an account but not identical local materialization timing; relay backup can re-introduce rows even when tombstones exist locally. Identity aliasing (local id vs Nostr event id) prevents reliable tombstone ↔ event alignment without a completed single materialization owner (R1 in refactor queue — **not** treated as in-flight for day-to-day feature work).
- **Product truth (ship as-is until refactor):**
  - **Delete for me** = hide on **this device / this session**; **not** guaranteed after refresh, restart, or profile switch.
  - **Same account on Web and Desktop** may show **different DM history** until backup and tombstones converge across devices.
  - Do **not** promise cross-surface delete durability in marketing or in-app success copy.
- **Engineering policy:** Do **not** spend further iteration on DM delete persistence, messaging storage layers, account-sync restore convergence, or database/object-model churn for this symptom unless a **scoped architecture milestone** is explicitly approved. Continue unrelated feature work.
- **Code note:** Partial mitigations remain in tree (`local-dm-visibility`, bootstrap seal, restore guardrails); they are **best-effort**, not a release guarantee.
- **Doc:** [`docs/messaging/deletion-roster-limitations.md`](../messaging/deletion-roster-limitations.md) § Accepted limitation.

### 2026-05-14 — **DM-001 (bootstrap re-import resurrection on refresh)** — partial fix (superseded by acceptance above)

- **Symptom:** Delete for me looked fine until **page refresh or dev-server restart**, then historical DM rows (e.g. 25× “test” from Tester2) all returned.
- **Root cause:** Account event log already had live `DM_RECEIVED` events but **no `BOOTSTRAP_IMPORT_APPLIED` marker**. On every load, `bootstrap()` ran **full legacy import** from chat-state/backup and appended **new** `DM_RECEIVED` rows **after** earlier `DM_REMOVED` events → replay resurrected messages. Tauri tombstone set was also read **before** SQLite hydrate during full bootstrap.
- **Fix:** If `existingEvents.length > 0` and marker missing → **`buildBootstrapSealEvents`** (tombstone `DM_REMOVED` + marker only, no chat-state re-import). Full `buildBootstrapAccountEvents` only on empty log. Bootstrap path hydrates SQLite tombstones with `profileId` before reading suppressed ids. Batch delete projection: single `appendCanonicalEvents` + one `replay`.
- **Verify:** Delete 25 messages → refresh → restart dev server → thread stays empty (one-time seal may append marker on first load after upgrade).

### 2026-05-14 — **DM-001 (delete survives reload — projection + tombstones)** — partial fix

- **Symptom:** Delete for me still lost on page refresh, dev-server restart, or any full reload; desktop and web IDB/SQLite split made triage hard.
- **Root cause:** DM read truth is **account projection** (IndexedDB event log). Deletes only suppressed SQLite/IDB but **`appendCanonicalDmRemovedEvent` was fire-and-forget (`void`)**, so `DM_REMOVED_LOCALLY` often never landed before reload; projection replay re-hydrated `DM_RECEIVED` rows. Startup replay also ran **before** SQLite tombstones were hydrated, so `isMessageDeleteSuppressed` missed on first paint.
- **Fix:** `applyDmDeleteForMePersistence` — await durable stores + **await** all `DM_REMOVED_LOCALLY` appends + `accountProjectionRuntime.replay`; wired from `use-chat-actions`; hydrate tombstones before projection bootstrap/replay in `account-projection-runtime.ts`.
- **Verify:** Desktop dev → delete incoming test messages → hard refresh and restart `pnpm dev:desktop`; thread should stay empty.

### 2026-05-14 — **DM-001 (desktop delete-for-me / SQLite)** — partial fix

- **Symptom:** Tester1 on **Tauri desktop** could not durably delete historical DM rows (often synced while Tester2 used Chrome); web IndexedDB vs desktop SQLite made debugging confusing; deletes appeared to work in-session but SQLite/IDB/chat-state could resurrect rows.
- **Root cause:** Delete-for-me in `use-chat-actions` only updated in-memory tombstones + single-scope chat state; SQLite tombstone flush and hard-delete were **async/fire-and-forget**; desktop shell still shares **IndexedDB** at the same origin, so IDB rows could re-merge on hydrate.
- **Fix:** Canonical `persistDmLocalDeleteIdentities` (`dm-local-delete-persistence.ts`) — await SQLite hydrate + tombstone flush + `dbDeleteMessages` + tombstones on desktop, purge IDB aliases on desktop, clear chat-state across scopes; wired from `deleteMessageForMe` and `MessagePersistenceService.deleteMessage`; flush queue applies SQLite deletes for batched v2 deletes; conversation hydrate awaits SQLite tombstones before first load on desktop.
- **Verify:** Desktop dev → multi-select incoming “test” messages → Delete for me → reload window; rows should stay gone.

### 2026-05-13 — **DM-001 (cold hydrate suppression)** — partial fix

- **Symptom:** “Delete for me” + **page refresh** still showed messages; tombstones existed but UI rehydrated from IndexedDB/authority without applying the durable suppressed-id set when **`liveMessages`** was empty.
- **Root cause:** `assembleDmHydrateThreadReadModel` applied `filterMessagesBySuppressedIds` only inside the `liveMessages.length > 0` branch.
- **Fix:** Always compute `hydratedSuppressed` and use it as the base `finalMessages` and as the merge input for live overlay (`dm-conversation-hydrate-read-model.ts`). Regression: `dm-conversation-hydrate-read-model.test.ts`.
- **Still verify on client:** bulk delete for me → refresh; with Account Restore banner, wait for completion and refresh again.

### 2026-05-13 — **DM-001 (batch “Delete for me”)** — partial fix

- **Symptom:** After multi-select **Delete for me**, a **full page refresh** restored messages (especially with **Account Restore** / relay replay). Single-message delete was less affected.
- **Root cause:** `ChatView` batch handler fired `onDeleteMessageForMe` in a `forEach` **without awaiting** async work, and the `finally` block ran **immediately** — concurrent `chatStateStoreService.removeMessageIdentities` updates could race, and persistence/tombstone work could still be in flight when the UI cleared batch mode.
- **Fix:** `handleBatchDeleteForMe` / `handleBatchDeleteForEveryone` are now **async**, use a **sequential `await`** loop, and only clear batch UI state in `finally` **after** all handlers settle. Props allow `void | Promise<void>` return types.
- **Files:** `chat-view.tsx`, `chat-view.test.tsx` (messaging components)
- **Still open:** **DM-001** / restore paths if messages reappear from **IndexedDB / projection** without consulting tombstones; re-test after restore completes.
