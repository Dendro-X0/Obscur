# v1.5.0 — Architecture refactor queue (execution order)

**Last updated:** 2026-05-15 — **R0/R1/R2** ClientGateway exits (ESLint + `pnpm gateway:boundaries:check`). **Phase 3 M1/M4** ingress→coordinator subscriber landed; M5 runtime gates open.  
**Purpose:** Ordered backlog for **structural** work after user smoke (A/B) already exposed real defects. Prefer **subtracting parallel paths** and **contract-first boundaries** over growing Vitest surface while specifications are still moving.

**Verification and checkpoints:** Until the refactor exit criteria are met, **client and manual A/B testing are deferred** as completion gates. After each checkpoint, update **`docs/handoffs/current-session.md`**, append to **`docs/program/v1.5.0-refactor-checkpoints.md`**, and follow **`docs/program/v1.5.0-refactor-verification-and-docs-policy.md`**.

**Policy (until materialization is unified):**

- Do **not** add broad new test suites as the primary deliverable; use **narrow contract checks** only where they lock a **completed** refactor slice.
- Each slice below should end with: **one owner**, **explicit `profileId` scope**, and **diagnostics** at the new boundary (`logAppEvent` or existing `obscurAppEvents`).

## R0 — Client unified gateway (**exit — engineering**)

**Problem:** Web / desktop / mobile and delete / sync / roster each had parallel entry points (`getResolvedStoragePorts`, `localDmVisibilityOwner`, `isTauri` in features).

**Target:** One `ClientGateway` per active profile (`@dweb/client-gateway` + `getResolvedClientGateway()`). Platform adapters install once in `ProfileRuntimeProvider`.

**Landed (2026-05-14):**

- Package `packages/dweb-client-gateway` — contracts, `buildClientGateway`
- `apps/pwa/app/features/runtime/services/client-gateway-adapter.ts`
- `apps/pwa/app/features/profiles/services/resolve-client-gateway.ts`
- Feature routes migrated for `messageDeleteTombstones` + `localDmVisibility` (DM delete, hydrate, projection, restore, backup)

**Exit:** ESLint + `pnpm gateway:boundaries:check`; typed `@dweb/client-gateway/community-roster` + `dm-materialization` + `messaging-diagnostics`. Runtime symptom gates (DM-001/MEM-001) remain **accepted limitations**.

**Doc:** `docs/gateway/client-unified-gateway.md`

## Observed defects → architectural root

| Symptom (user smoke) | Root cause class | Refactor slice |
| --- | --- | --- |
| “Delete for me” messages return in a **new window** | Multiple hydrators (IndexedDB window, `chatStateStore`, account **projection** replay, relay replay) without a **single** suppression + identity pass | **R1** — Message list materialization |
| Community member list **collapses to one** after refresh | Roster/directory/snapshot **multiplicity** + warm-up policy vs persisted OR-set | **R2** — Community visible-members single source |

## R1 — Single message materialization boundary (highest leverage)

**Today (multiplicity):** `use-conversation-messages` (IDB + tombstone ref), `account-projection-selectors` / reducer (projection timeline), `chat-state-store` (persisted fallback), native SQLite tombstones (desktop), relay replay paths may each introduce rows with **different id aliases** for the same event.

**Target:**

1. One **read model** function (or small service) per `(profileId, conversationId)` that: loads suppression set (all aliases), merges sources in **fixed order**, and outputs displayable rows. Call sites become thin.
2. **Phase 2 alignment:** expose that read model behind a `@dweb/storage-contracts` port when stable; PWA IndexedDB + native SQLite adapters stay the only engines.
3. **New window / restore:** contract that **suppression + `removedMessageIds`** are applied **before** treating relay or projection replay as authoritative for visibility (ordering is part of the same slice).

**Interim milestones landed (still not R1 exit):**

1. **Hydrate authority + legacy count gates** — single module: `apps/pwa/app/features/messaging/services/dm-read-authority-contract.ts` (`resolveHydrationDmReadMessages`, `resolveLegacyHydrationAuthority`, repair predicates; former `conversation-history-authority*.ts` removed or folded).
2. **Authority-selected merges / caps + hydrate overlay assembly** — `apps/pwa/app/features/messaging/services/conversation-message-materialization.ts` + **`dm-conversation-hydrate-read-model.ts`** (`assembleDmHydrateThreadReadModel`: `resolveHydrationDmReadMessages`, live overlay merge, caps, log payloads for authority diagnostics).
3. **Projection evidence → live message list merge** — `apps/pwa/app/features/messaging/services/dm-conversation-projection-live-merge.ts` (`mergeProjectionFirstWithLiveOverlayForDisplay`, `areMessageListsEquivalentById`; hook projection merge `useEffect`).
4. **DM row → `Message` normalize** — `apps/pwa/app/features/messaging/services/dm-conversation-normalize-message.ts` (`normalizeDmConversationMessageRow`; IndexedDB hydrate, projection evidence, realtime **`applyBufferedEvents`**).
5. **DM displayable line predicate** — `apps/pwa/app/features/messaging/services/dm-conversation-displayable-message.ts` (`isVoiceCallSignalPayload`, `isDisplayableDmConversationMessage`; hook + **`assembleDmHydrateThreadReadModel`** group-scope filter).
6. **DM hydrate orchestration (pre–full R1 exit)** — `apps/pwa/app/features/messaging/services/dm-conversation-hydrate-pipeline.ts` — **`runDmConversationHydrateReadModelPipeline`** (tombstone prep → Indexed scan → persisted chat-state fallback → **`assembleDmHydrateThreadReadModel`** → sibling id-split diagnostics) + **`logDmHydrateReadModelTelemetry`**; **`use-conversation-messages`** `hydrateHistory` delegates to this module (React state/refs stay in the hook).

**Primary files to converge (starting points, not final owner):**

- `apps/pwa/app/features/messaging/hooks/use-conversation-messages.ts` (tombstone ref prep before hydrate, **`normalizeDmConversationMessageRow`** + **`isDisplayableDmConversationMessage`** wiring, live overlay merge, projection snapshot assembly)

## R2 — Community visible members: single durable OR-set

**Today:** `group-provider` merges roster projection, known-participant directory, and `useSealedCommunity` / relay snapshots; snapshot policy can still race refresh. **Persist path:** durable `upsertCommunityKnownParticipantsEntry` runs only when the built directory widens beyond stored ∪ `group.memberPubkeys` ∪ local (no parallel `createdGroups`→storage upsert).

**Target:**

1. One **durable** “known participants” projection keyed by `(profileId, conversationId)` with **monotonic union** semantics for pubkeys until explicit leave/expel evidence.
2. UI reads only that read model + terminal membership ledger; relay snapshots **narrow** only when evidence confidence is high (existing policy direction).

**Primary files:**

- `apps/pwa/app/features/groups/providers/group-provider.tsx`
- `apps/pwa/app/features/groups/services/community-known-participant-directory.ts`
- `apps/pwa/app/features/groups/services/community-member-snapshot-policy.ts`
- `apps/pwa/app/groups/[...id]/group-home-page-client.tsx` (stabilization via **`useStableCommunityParticipantPubkeys`** — shrink further once R2 read model is canonical)

**Exit (engineering):** `communityRoster` gateway port + package contract; group home, management, and main-shell chat header use R2 read model. **MEM-001** runtime proof deferred to batched manual matrix.
