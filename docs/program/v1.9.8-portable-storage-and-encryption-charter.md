# v1.9.8 — Portable storage paths & encryption-at-rest charter

**Status:** Draft product charter (2026-06-13)  
**Builds on:** [production-surfaces.md](../architecture/production-surfaces.md) · prior UX iteration (export/archive clarity)  
**Audience:** Maintainers, security review, business / portable-device users

---

## 1. Product intent

Obscur should behave like a **self-contained workspace** the user owns — not a second hidden tree on the OS drive beside the installer.

| User story | Requirement |
|------------|-------------|
| **Portable business use** | Point all durable user data (profiles, messages, vault, exports) at a folder on removable or synced storage; move devices by moving that folder. |
| **No plaintext surprise** | Files under that folder must not reveal private content to anyone who only has filesystem access. Reading content requires **unlocking Obscur** with the device login password (or equivalent credential entered in the client). |
| **Vault trust** | Cached / local media in **Vault** must meet the same bar as messages — filenames, thumbnails, and bytes must not leak identity or content offline. |
| **Honest limits** | While the app is **unlocked**, a sophisticated attacker with live remote control may read memory or screen — that is out of scope for “filesystem-only” protection. |

This charter defines **what we guarantee**, **what exists today**, and **what to build next**.

## 2. Storage layout model (target)

### 2.1 Three path classes

Users configure paths in **Settings → Storage** (not scattered across Advanced panels).

| Class | Default | User override | Contents | Portable? |
|-------|---------|---------------|----------|-------------|
| **User data root** | OS app-data (fallback) | **Yes — primary control** | Per-profile SQLite, vault blobs, identity markers, `workspace-exports/`, `profile-archives/`, profile registry pointer | **Yes** — this is the folder users move |
| **Cache root** | Subfolder of user data or OS cache | Optional | Thumbnails, decompressed previews, relay temp, WebView cache (if relocatable) | No — safe to delete |
| **Runtime root** | Install dir / OS temp | Usually fixed | Logs, updater staging, Tor child (desktop) | No |

**Rule:** Only **user data root** must be relocatable for the portable-device story. Cache and runtime may stay on fast local disk even when user data is on removable media.

### 2.2 Single tree the user recognizes

```
{UserDataRoot}/
  obscur.json                 # manifest: version, profile list, path class metadata
  profiles/
    {profileId}/
      workspace.db            # encrypted SQLite (target)
      vault/                  # encrypted media blobs (target)
      webview/                # browser profile (OS-managed; minimize sensitive data here)
  workspace-exports/          # user-initiated encrypted backups (.obscur-bundle)
  profile-archives/           # removal snapshots (target: encrypted or metadata-only)
```

Today: **partial** — `resolve_effective_data_root()` in `apps/desktop/src-tauri/src/data_root.rs` relocates exports/archives and profile dirs, but UI exposes one “Obscur Data Directory” and vault can use a **separate** `customRootPath` in `local-media-store.ts`.

**Bind contract:** [obscur-data-root-bind-contract.md](./obscur-data-root-bind-contract.md) — anchor vs physical root, redirect (junction/symlink), pointer fallback, heal.

**v1.9.8 UX goal:** One **“Your Obscur data folder”** control + optional advanced split (cache only).

## 3. Encryption-at-rest — threat model

### 3.1 Scenarios

| ID | Scenario | Target outcome |
|----|----------|----------------|
| **T1** | User browses `%AppData%` or removable drive in Explorer | Sees opaque files / encrypted DB; no readable message bodies or media |
| **T2** | Laptop stolen, powered off, attacker copies disk | Without device login password, decryption infeasible (keys not stored plaintext) |
| **T3** | Attacker copies `workspace-exports/*.obscur-bundle` | Already encrypted with account key — **pass today** for unified exports |
| **T4** | Attacker copies `profile-archives/*.obscur-profile.json` | **Fail today** — plaintext scoped localStorage snapshot |
| **T5** | Attacker copies `vault-media/*` with original filenames | **Fail today** — plaintext files on disk |
| **T6** | Attacker copies native SQLite | **Fail today** — not SQLCipher; readable with standard tools |
| **T7** | Remote control while app **locked** | No key material in app process; files remain T1/T2 |
| **T8** | Remote control while app **unlocked** | **Best-effort** — user session is compromised; document limit |

### 3.2 Credential → keys (canonical approach)

Do **not** invent non-standard crypto. Use a well-studied **envelope**:

1. **Device login password** (user-entered at unlock) + per-profile **salt** → **Profile Data Key (PDK)** via Argon2id (or scrypt).
2. **PDK** wraps:
   - SQLite encryption key (SQLCipher PRAGMA key or equivalent)
   - Vault **master wrap key** (KW)
   - Optional export of wrapped keys for backup bundles
3. **PDK lives only in memory** while unlocked; zeroized on lock / process exit.
4. **Stay signed in / OS keychain** may store a **wrapped PDK** or unlock token — never the plaintext password. Compromise of keychain alone must not equal full plaintext access without further user action (document platform limits).

**Alternative considered:** OS full-disk encryption only — **insufficient** for product claim; does not protect against second user on same machine or remote filesystem access.

**“Plaintext does not expose private information”** fallback: acceptable only for **non-sensitive** metadata (e.g. opaque UUID filenames, aggregate counts). Message bodies, contact graph, vault bytes, and key material must not rely on this alone.

## 4. Component map — today vs target

| Component | Today | v1.9.8 target | Owner (proposed) |
|-----------|-------|---------------|------------------|
| **Data root picker** | Settings → Storage → “Obscur Data Directory”; restart required | Same + manifest `obscur.json`; validate portable drive | `data_root.rs` + storage settings panel |
| **Unified export** | Encrypted `.obscur-bundle` | Keep; document as “encrypted backup” | `unified-account-export-service.ts` |
| **Removal archive** | Plaintext `.obscur-profile.json` | Encrypt with PDK or reduce to non-sensitive tombstone | `profile-workspace-archive-service.ts` |
| **Native SQLite** | Plaintext per profile | SQLCipher or Rust-side encryption layer | `@dweb/db` + Tauri init |
| **Vault local cache** | Plaintext files under `vault-media/` | Per-file AEAD (`obscur-file-aead-v1`); opaque paths | `local-media-store.ts` + native adapter |
| **Vault index** | localStorage JSON (paths, names) | Encrypt index or store inside SQLite | `local-media-store.ts` |
| **IndexedDB message queue** | Optional `encryptStorageAtRest` (PWA legacy) | Native path ignores this — **do not claim** encryption from that toggle on desktop | `message-queue.ts` (web only) |
| **WebView profile dir** | May hold readable cache | Minimize; prefer SQLite + encrypted vault | `profiles.rs` `data_directory` |

## 5. Vault-specific study (required before implementation)

Vault is the highest-risk surface for **T1/T2** because files are large and historically stored with **readable names**.

### 5.1 Questions to answer

1. **Encrypt on write or encrypt on cache?**  
   - On write: every local copy is ciphertext immediately.  
   - On cache: decrypt only in memory for display; disk always ciphertext.

2. **Thumbnail strategy**  
   - Encrypted sidecar vs generated in-memory only vs low-res encrypted blob.

3. **Dedup / CAS** (`media-cas-store.ts`)  
   - Content hashes leak file identity — are hashes of **ciphertext** acceptable?

4. **Performance on removable USB**  
   - SQLCipher + random-access vault on slow media — soak tests required.

5. **Backup include vault**  
   - Already optional in workspace bundle (size caps). Align caps with encrypted vault layout.

### 5.2 Recommended default (pending soak)

- **File format:** `{vault}/{opaqueId}.obscurvault` — AEAD blob; metadata in encrypted SQLite row.
- **Key:** Per-file content key wrapped by KW derived from PDK.
- **Display name:** Only inside encrypted DB / index — never in filename.

## 9. Non-goals (v1.9.8)

- Cloud sync of data root
- Hardware security module requirement
- Protection against unlocked-session malware (T8) beyond normal lock discipline
- Replacing account identity cryptography (Nostr keys) — this charter is **local persistence** only

---

## 10. Exit criterion for “encryption guarantee” marketing copy

All must pass on **desktop native** with user data on a **non-default path**:

1. `workspace.db` — not readable with `sqlite3` CLI without unlock session
2. Vault files — no plaintext media; filenames non-descriptive
3. Removal archives — encrypted or contain no message/vault payload
4. Lock → verify no PDK in crash dumps / documented key lifecycle
5. Unified export — unchanged encrypted behavior

Until then, product copy: **“Exports are encrypted; local database and vault encryption — in progress (v1.9.8).”**
