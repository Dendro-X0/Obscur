# v1.9.6 — Chrome-style profile picker charter

**Status:** Slice 1–2 landed  
**Target release band:** v1.9.6  
**Canonical owner:** profile window lifecycle (`desktop-profile-runtime`, `AuthGateway`)

## Problem

Today, **Manage Profiles** lived behind an unlocked session. A locked desktop window showed a bound-account login form first; users could not browse, add, or open profile windows without signing into the window’s current profile.

That blocked the Chrome mental model: **peer windows**, profile picker at startup, sign in to whichever profile you choose.

## Goals

1. **Public profile management** — `/profiles` reachable without account unlock on desktop.
2. **Independent windows** — any profile can open in its own window; Window 2 does not require Window 1.
3. **Clear sign-in path** — from the picker, sign in / switch / open in new window.
4. **No regression to unlocked UX** — signed-in users keep full app chrome + title-bar switcher.

## Architecture

| Layer | Change |
|-------|--------|
| `auth-public-routes.ts` | Public routes + redirect policy (respects show-on-startup) |
| `profile-picker-startup-policy.ts` | Device-level “Show on startup” preference |
| `AuthGateway` | Render children on public routes; redirect locked `/` → `/profiles` when enabled |
| `ProfilePickerChromeHost` | Minimal title bar on locked `/profiles` (no account badge) |
| `ProfilePickerCardGrid` | Chrome-style “Who’s using Obscur?” card grid |
| `profiles/page.tsx` | Grid default when locked; advanced management for remove/backup |
| `providers.tsx` | Title bar inside provider tree (context-aware) |
