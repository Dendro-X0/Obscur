## 1. Problem statement

### Symptoms (user-visible)

| Symptom | Root cause class |
|---------|------------------|
| A leaves; B still shows A as active member | Cross-client membership evidence not delivered or not applied as authority |
| Re-invite blocked after leave | Membership projection conflated with discovery/history |
| Delete-for-me messages return | Multiple hydrators without single suppression owner (R1) |
| Roster collapses to one member after refresh | Parallel roster paths + relay snapshot races (R2) |
| App feels “unfinished” | Application protocol truth implemented inside Nostr kind handlers and React state |

### Architectural violation

**Relay-visible Nostr kinds** (9022, 39002, etc.) are treated as **membership truth**. Protocol 22 requires:

> Relays are transport only. Membership truth is reducer-driven and evidence-backed.

## 2. Target architecture

```text
┌──────────────────────────────────────────────────────────────────┐
│ APPLICATION (apps/pwa, apps/desktop)                                │
│  UI · settings · transport/membership mode selection              │
└────────────────────────────┬─────────────────────────────────────┘
                             │ ClientGateway + kernel ports
┌────────────────────────────▼─────────────────────────────────────┐
│ KERNEL (packages — v1.9 introduces/expands)                       │
│  • CommunityMembershipPort — join/leave/expel reducer + ledger    │
│  • DmMaterializationPort — suppression + display read model (R1)  │
│  • CommunityRosterPort — monotonic discovery + terminal set (R2) │
│  • Semantic event types (not Nostr kinds)                         │
└────────────────────────────┬─────────────────────────────────────┘
                             │ TransportPort
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        NostrAdapter   CoordinationAdapter  (future TeamRelay)
        (kinds→events) (HTTP/WS head+delta)  (private fanout)
              │              │
              └──────┬───────┘
                     ▼
              Relay / network (non-authoritative)
```

### Layer rules (enforced in v1.9)

1. **Features** (`apps/pwa/app/features/**`) do not import `@dweb/nostr` for new code; ESLint enforced in B0.
2. **Nostr kinds** are parsed only inside `packages/dweb-transport-nostr` (new or extended package name — see §4.1).
3. **Membership UI** reads `CommunityRosterPort` + `CommunityMembershipPort` only — not `useSealedCommunity` state directly for invite/send gates.
4. **Coordination** stores no message plaintext, no room keys — only signed membership metadata and relay hints.

## 3. Semantic event model (kernel input)

All transports map wire payloads to **semantic events** before reducers run.

### Community membership events

| Semantic type | Source examples | Reducer effect |
|---------------|-----------------|---------------|
| `COMMUNITY_MEMBER_JOINED` | invite accept, restate, coordination delta | Add to active set |
| `COMMUNITY_MEMBER_LEFT` | sealed leave, coordination delta, local leave intent | Terminal + remove from active |
| `COMMUNITY_MEMBER_EXPELLED` | governance, coordination | Terminal |
| `COMMUNITY_DIRECTORY_WIDEN` | message author, relay hint 39002 | Discovery only — **never** shrinks |

### Control vs content (protocol 22)

- **Community Control Envelope** — membership, governance, leave (B1/B2).
- **Community Content Envelope** — sealed chat (existing 10105 path via adapter).
- Room-key distribution stays **private direct envelope** (protocol 23).

#### 4.2 TransportPort (contract sketch)

```typescript
// packages/dweb-transport-contracts — illustrative; implement in B0
export type TransportKind = "nostr" | "obscur_coordination" | "team_relay";

export type SemanticCommunityEvent =
  | { type: "COMMUNITY_MEMBER_LEFT"; communityId: string; subjectPubkey: string; createdAtUnixMs: number; logicalEventId: string; source: TransportKind }
  | { type: "COMMUNITY_MEMBER_JOINED"; /* ... */ }
  | { type: "COMMUNITY_DIRECTORY_HINT"; pubkeys: readonly string[]; confidence: "hint" };

export interface TransportPort {
  readonly kind: TransportKind;
  publishCommunityControl(event: SemanticCommunityEvent): Promise<PublishResult>;
  subscribeCommunityControl(handler: (event: SemanticCommunityEvent) => void): Unsubscribe;
}
```

### B2 — Coordination directory (`v1.9.2`)

**Goal:** Reliable cross-client membership fanout without trusting public relay roster.

#### 4.7 Coordination service API (extends `apps/coordination`)

Base URL configured per deployment (`OBSCUR_COORDINATION_URL`). All bodies signed by community steward or member key per endpoint policy.

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/health` | GET | Existing |
| `/invites/create` | POST | Existing |
| `/invites/redeem` | POST | Existing |
| `/communities/:id/membership/head` | GET | Returns latest seq, head hash, signer pubkeys |
| `/communities/:id/membership/delta` | POST | Append signed delta (join/leave/expel) |
| `/communities/:id/membership/subscribe` | WS (or SSE) | Push deltas to connected clients |

**Delta payload (signed, no plaintext chat):**

```typescript
type MembershipDelta = Readonly<{
  communityId: string;
  seq: number;
  action: "join" | "leave" | "expel";
  subjectPubkey: string;
  actorPubkey: string;
  createdAtUnixMs: number;
  signature: string; // schnorr over canonical JSON
}>;
```

#### 4.8 Client wiring

- Implement `CoordinationTransportAdapter` implementing `TransportPort`.
- Replace TODO in `community-membership-sync.ts` with coordination subscribe → semantic events → B1 port.
- Feature flag: `membershipSyncMode: "nostr_only" | "coordination_preferred"`.

**Default for new communities on public relay:** `coordination_preferred` when `OBSCUR_COORDINATION_URL` set; else `nostr_only` with honest UI.

## 5. Migration & compatibility

| Concern | Policy |
|---------|--------|
| Existing communities | On upgrade, hydrate ledger from local state; coordination head optional backfill via reconcile |
| Nostr-only users | Full functionality with **documented limits** — no coordination required |
| Other Nostr clients | Not required to understand Obscur coordination; interop is best-effort on open kinds |
| Feature flags | `transportMode`, `membershipSyncMode` — default safe (nostr_only) until B2 tested |
