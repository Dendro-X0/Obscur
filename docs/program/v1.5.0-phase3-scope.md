## Problem Statement
Community membership currently has overlapping mutation paths, including DM-driven side effects that are not durable network truth. This causes sender/receiver divergence, restart inconsistencies, and fragile convergence.

Phase 3 fixes this by enforcing one canonical membership owner path based on relay evidence and deterministic merge semantics.

## In Scope
- Relay-first membership ingest for join/leave/disband/snapshot evidence.
- Typed domain events and profile-scoped routing at ingress boundaries.
- Single canonical membership mutation owner (coordinator/ledger path).
- Deterministic merge semantics for concurrent evidence.
- Invite accept/decline flow alignment with canonical membership evidence.
- Focused A/B correctness tests (same-process + two-client reasoning).
- Diagnostics coverage for ingress -> classify -> mutate pipeline.

## Out of Scope
- New social/product features unrelated to membership correctness.
- Full Phase 4 integration concerns (Tor system-wide verification, full multi-profile UX).
- Broad storage-port expansion beyond what Phase 3 directly needs.
- Large refactors without direct membership convergence impact.

## Canonical Owner Contract
- Membership state is updated only from evidence-backed protocol events.
- UI may render provisional states, but must not commit authoritative membership from local-only optimism.
- Any legacy DM-driven membership mutation must be removed or quarantined behind non-authoritative surfaces.
