# 26 v1.0.7 CP3 Community Convergence Matrix

_Last reviewed: 2026-03-23 (baseline commit 4c869a7)._

This matrix defines the manual two-device/account-switch replay evidence required before `v1.0.7` release preflight/tagging (`CP3` in `docs/25-versioned-phase-plan-v1.0.7-v1.0.9.md`).

## Scope

1. Community identity continuity (name/metadata does not regress to placeholders).
2. Membership continuity across account switch + restart.
3. Sendability continuity (room-key and ledger convergence under restore replay).
4. Convergence diagnostics quality for reproducible incident bundles.

## Session Setup

1. Use one desktop runtime and one web runtime.
2. Prepare two test accounts:
: `A` (community creator/operator),
: `B` (joined member).
3. Ensure at least one community has:
: a non-placeholder custom display name,
: `A` and `B` in member coverage,
: recent message activity from `A` and `B`.
4. Keep browser console open for copy commands.

## Replay Steps

1. Clear app-event buffer on active runtime:
: `window.obscurAppEvents?.clear()`
2. On account `A`, verify baseline before switch:
: community name is custom (not `Private Group`),
: members include `A` and `B`,
: `A` can send a community message.
3. On the same runtime, switch account sequence:
: `A -> B -> A`.
4. After switching back to `A`, verify:
: community still appears in Network list,
: community name remains custom,
: member list still includes `A`,
: `A` can send without `No room key found` errors.
5. Restart runtime while still on `A` and repeat step 4.
6. Verify cross-device continuity:
: `B` receives new `A` message,
: `B` can still send to the same community.
7. Export CP3 evidence bundle:
: `copy(JSON.stringify((() => {`
: `  const digest = window.obscurAppEvents?.getCrossDeviceSyncDigest?.(400);`
: `  return {`
: `    generatedAtUnixMs: Date.now(),`
: `    communityLifecycleConvergence: digest?.summary?.communityLifecycleConvergence ?? null,`
: `    membershipSendability: digest?.summary?.membershipSendability ?? null,`
: `    membershipRecoveryHydrate: digest?.events?.["groups.membership_recovery_hydrate"] ?? [],`
: `    membershipLedgerLoad: digest?.events?.["groups.membership_ledger_load"] ?? [],`
: `    roomKeyMissingSendBlocked: digest?.events?.["groups.room_key_missing_send_blocked"] ?? [],`
: `    recentWarnOrError: digest?.recentWarnOrError ?? [],`
: `    m0Triage: window.obscurM0Triage?.capture?.(300) ?? null,`
: `  };`
: `})(), null, 2))`

## Pass Criteria

1. No community metadata regression to placeholder name after switch/restart.
2. `A` remains in member coverage after switch/restart.
3. `A` and `B` both can send in the target community.
4. `communityLifecycleConvergence.riskLevel` is not `high`.
5. `roomKeyMissingSendBlocked` event count remains `0` for the replay window.

## Escalation Criteria

Escalate as active blocker if either occurs:

1. Community name regresses to placeholder/default or creator membership disappears after switch/restart.
2. Any `groups.room_key_missing_send_blocked` event appears during replay.
3. `communityLifecycleConvergence.riskLevel === "high"`.

When escalating, attach:

1. CP3 evidence bundle from step 7.
2. `copy(window.obscurM0Triage?.captureJson(300))`
3. Short timestamped replay timeline (`A -> B -> A`, restart point, first failure action).
