## Purpose

This plan exists to prevent another round of scattered fixes for community membership resurrection, profile-scope drift, and restore visibility regressions.

The product goal is user-facing decentralization and data sovereignty: users own identity, keys, private state, backup/export continuity, and privacy boundaries.

The architecture goal is not internal decentralization. The application should use coordinated product ownership with modular implementation. Modules should be easy to maintain, extend, and integrate, but they must not independently decide the same membership truth.

## Problem Statement

Community membership state is currently affected by multiple semi-independent paths:

- `group-provider.tsx` mutates visible groups, chat-state groups, roster projection, known participants, and membership ledger entries.
- `community-membership-recovery.ts` reconstructs visible groups and produces fallback `joined` ledger backfills.
- `community-membership-ledger.ts` owns persistence, legacy/scoped storage fallback, normalization, merge precedence, validation, and event dispatch.
- Restore materialization writes restored ledger and chat-state snapshots separately.
- Browser-global runtime events can trigger membership mutations outside a single canonical action path.

This dispersion allowed local fixes to conflict. A path that preserves restore can resurrect a left community. A path that blocks resurrection can break legitimate fresh-device restore. A profile-scope storage bridge can fix bootstrap visibility while risking cross-profile leakage.

The fix must consolidate membership decisions into one product-owned coordinator while keeping persistence, restore, runtime event adapters, and UI projection modular.

## Non-Goals

Do not use this work to pursue backend-level decentralization for its own sake.

Do not create a new independent subsystem that competes with the provider, ledger, restore, or sealed-community runtime.

Do not keep adding call-site guards such as `isExplicitlyLeft` across every event handler as the primary solution.

Do not treat `createdGroups`, chat-state rows, relay snapshots, or backup payloads as membership truth by themselves.

## Target Principle

Decentralize user power, not internal product authority.

Users should retain sovereignty over their data and identity. Internally, the app should have one coordinated membership decision owner with narrow, explicit adapters around it.

## Canonical Owner

Introduce or designate a pure community membership coordinator, likely:

`apps/pwa/app/features/groups/services/community-membership-coordinator.ts`

The coordinator is the only owner that answers:

- Is this community visible for this public key/profile?
- What is the user's current membership status?
- Which evidence wins when sources disagree?
- Is a ledger mutation allowed?
- Is a restore fallback allowed to infer joined membership?
- Is this an explicit revive/rejoin or an accidental resurrection?

## Coordinator Inputs

The coordinator must receive explicit inputs. It must not read ambient identity, ambient profile scope, or localStorage directly.

Required inputs:

- `publicKeyHex`
- `profileId`
- persisted chat-state groups
- membership ledger entries
- group tombstones
- known participant evidence if needed for display
- restored backup evidence if applicable
- runtime event evidence if applicable
- relay/roster/leave evidence if applicable

## Coordinator Outputs

The coordinator must return decisions rather than performing storage writes directly.

Expected outputs:

- visible `GroupConversation` projection
- membership projections
- descriptor projections if needed by existing surfaces
- allowed ledger mutations
- suppressed mutation diagnostics
- visibility diagnostics
- restore/revival classification

Example shape:

```ts
export type CommunityMembershipCoordinatorResult = Readonly<{
  groups: ReadonlyArray<GroupConversation>;
  membershipProjections: ReadonlyArray<CommunityMembershipProjection>;
  descriptorProjections: ReadonlyArray<CommunityDescriptorProjection>;
  ledgerMutations: ReadonlyArray<CommunityMembershipLedgerMutation>;
  diagnostics: CommunityMembershipCoordinatorDiagnostics;
}>;
```

The first implementation should be pure and testable without React, browser events, or localStorage.

## Evidence Classes

Stop treating all `joined` writes as equal.

Define evidence classes so precedence can be reasoned about explicitly:

```ts
export type CommunityMembershipEvidenceKind =
  | "user_explicit_leave"
  | "user_explicit_join"
  | "user_explicit_rejoin"
  | "runtime_membership_confirmed"
  | "runtime_invite_accepted"
  | "restore_ledger_snapshot"
  | "restore_chat_state_group"
  | "restore_room_key_invite_accept"
  | "relay_roster_snapshot"
  | "relay_leave_event"
  | "historical_descriptor";
```

## Precedence Rules

### Rule 1: Explicit local leave wins

`left` or `expelled` from explicit user action must beat:

- persisted chat-state groups,
- backup fallback groups,
- historical descriptors,
- missing ledger coverage repair,
- ambient membership-confirmed events without fresh user intent.

### Rule 2: Revival must be named

A previously left community can become joined only through an explicit revive path:

- user clicks join/rejoin,
- user accepts a new invite with matching current account identity,
- another future explicit user-controlled recovery path.

Ambient refresh, login, restore, or historical evidence must not revive.

### Rule 3: Restore can infer only absent state

Restore/chat-state fallback may infer `joined` only when there is no explicit `left` or `expelled` evidence for that membership key.

### Rule 4: UI is projection, not truth

`createdGroups` is not canonical. It is a rendered projection of coordinator decisions.

### Rule 5: Ledger writes require coordinator approval

No caller should convert a group row into a `joined` ledger entry without coordinator evaluation.

## Module Boundaries

### `community-membership-ledger.ts`

Owns:

- storage key construction,
- scoped/legacy persistence mechanics,
- normalization,
- schema migration,
- primitive merge helpers,
- event emission after accepted writes.

Does not own:

- restore visibility policy,
- revival policy,
- UI projection policy.

### `community-membership-recovery.ts`

Should become either part of the coordinator or a pure evidence/projection helper called by the coordinator.

It should not independently decide durable membership truth.

### `group-provider.tsx`

Owns:

- React composition,
- active identity/profile wiring,
- calling coordinator with explicit inputs,
- rendering coordinator output into provider state,
- applying coordinator-approved mutations.

Does not own:

- membership precedence,
- resurrection prevention policy,
- restore fallback policy.

### `restore-materialization.ts`

Owns:

- applying restored private-state material to stores,
- emitting restore diagnostics.

Does not own:

- final community visibility,
- revival policy.

After restore writes inputs, the coordinator must re-evaluate projection deterministically.

## Profile-Scope Policy

Profile scope must be explicit. Bootstrap restore must not depend on accidental legacy fallback behavior.

Target policy:

- Restore materialization receives a `targetProfileId`.
- Membership ledger writes are scoped to that target profile.
- Legacy fallback writes are deliberate compatibility behavior, not the main bridge between bootstrap and default profile.
- Named profiles must not overwrite another profile's legacy/default membership state.

Long-term target API:

```ts
saveCommunityMembershipLedger(publicKeyHex, entries, {
  profileId: targetProfileId,
  writeLegacyFallback: targetProfileId === getDefaultProfileId(),
});
```

## Implementation Slices

### Slice A: Pure coordinator contract

Create the coordinator module and tests. No React. No localStorage. No browser events.

Required tests:

- left ledger beats persisted chat-state group,
- missing ledger allows restore fallback,
- explicit rejoin revives left state,
- runtime membership confirmed does not revive left state by default,
- restore ledger snapshot preserves explicit left over inferred joined.

### Slice B: Hydration path migration

Change `hydrateGroupsForPublicKey` to:

1. load inputs,
2. call coordinator,
3. set provider state from coordinator groups,
4. apply only coordinator-approved ledger mutations.

Remove direct missing-ledger `joined` backfills from the provider path unless produced by coordinator output.

### Slice C: Event path migration

Route these through coordinator decisions:

- invite accepted,
- membership confirmed,
- group add/rejoin,
- leave/remove,
- roster snapshot updates where membership status changes.

Browser-global events should stay identity-quarantined, but identity checks are an adapter guard, not the full membership policy.

### Slice D: Restore path synchronization

After restore materialization writes ledger/chat-state inputs, ensure the provider refresh uses coordinator output and does not depend on event ordering between ledger writes and chat-state writes.

### Slice E: Profile-scope cleanup

Replace implicit active-profile ledger reads/writes in critical restore paths with explicit `profileId` arguments.

Keep compatibility fallback only where documented and tested.

## Validation Gates

Do not call v1.5.0 complete until all of these are true:

- coordinator unit tests pass,
- ledger precedence tests pass,
- group-provider hydration tests pass,
- restore materialization tests pass,
- same-process A/B replay passes,
- runtime replay confirms no membership resurrection after leave,
- explicit rejoin still works,
- fresh-device restore still recovers legitimate joined communities,
- profile-scope rebinding does not lose restored groups or leak groups across profiles.

## Same-Process A/B Release Gate

Minimum replay:

1. A creates a community.
2. B joins through the canonical path.
3. B sees the community and messages.
4. B leaves.
5. B refreshes/relogs/restores.
6. B does not see the left community unless explicitly rejoining.
7. A eventually sees B removed from active membership/invite blocking.
8. B explicitly rejoins and visibility returns through the named revive path.

## Diagnostics Required

Coordinator decisions should emit or return enough diagnostic context to answer:

- which evidence sources existed,
- which evidence won,
- whether a mutation was suppressed,
- whether a joined state was inferred or explicit,
- whether a left/expelled state blocked resurrection,
- which profile scope was used.

Suggested diagnostic names:

- `groups.membership_coordinator_projected`
- `groups.membership_coordinator_mutation_allowed`
- `groups.membership_coordinator_mutation_suppressed`
- `groups.membership_coordinator_restore_inferred`
- `groups.membership_coordinator_revive_required`

## Current Known Risks

- `group-provider.tsx` currently owns too many lifecycle/state responsibilities.
- `community-membership-ledger.ts` mixes persistence, migration, validation, and merge precedence.
- `community-membership-recovery.ts` can infer joined fallback from persisted groups.
- Restore writes ledger and chat-state separately, so event ordering can affect visibility.
- Scoped/legacy ledger fallback can either fix bootstrap restore or create profile leakage if not explicit.
