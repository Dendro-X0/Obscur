# 38 v1.5 Community Runtime Owner and Projection Spec

_Last reviewed: 2026-05-03 (baseline commit 7111e10a)._

## Purpose

This spec defines the v1.5.0 target owner boundary for community runtime state. It prepares the implementation refactor by naming the single owner, the evidence contracts, the projection contracts, and the legacy paths that must become adapters or migration inputs.

## Non-Negotiable Rule

A community lifecycle action must have one canonical mutation path.

If existing code has multiple owners for the same state, v1.5.0 should remove or quarantine owners instead of adding another compatibility layer.

## Current Owner Drift

The current system spreads community truth across:

1. React provider state,
2. persisted chat-state groups,
3. membership ledger entries,
4. local tombstones,
5. CRDT membership containers,
6. sealed community event ingest,
7. restore merge reconstruction,
8. room-key snapshots,
9. browser-global events,
10. known participants storage.

This is the class of drift that produces ghost communities, self-only rosters, stale sendability, and cross-profile contamination.

## Target Owner Boundary

The v1.5.0 community runtime owner is responsible for these domains:

1. descriptor projection,
2. self-membership projection,
3. member roster projection,
4. room-key availability projection,
5. sendability projection,
6. private self-membership intent persistence,
7. community lifecycle mutation API,
8. static restore evidence ingestion,
9. live relay evidence ingestion,
10. relay publish outbox coordination.

React providers and UI components consume projections from the owner. They do not decide membership truth.

## Scope Contract

All calls into the runtime owner must carry explicit scope:

```ts
type CommunityRuntimeScope = Readonly<{
  profileId: string;
  publicKeyHex: string;
  deviceId: string;
  windowId: string;
}>;
```

### Scope Rules

1. Do not derive account/profile scope from module-level mutable state inside storage modules.
2. Do not use an active-user fallback when a caller can pass scope.
3. Do not emit unscoped community mutation events.
4. Do not write profile-scoped state to legacy account-global keys except through a named migration bridge.
5. Every diagnostic in this owner must include enough scope to distinguish accounts A and B in one process.

## Evidence Contract

```ts
type CommunityEvidenceSource =
  | "local_user_action"
  | "live_relay"
  | "restore_backup"
  | "legacy_migration"
  | "retry_outbox";

type CommunityEvidenceFreshness =
  | "current"
  | "historical"
  | "unknown";

type CommunityEvidence = Readonly<{
  source: CommunityEvidenceSource;
  freshness: CommunityEvidenceFreshness;
  communityId: string;
  groupId: string;
  relayUrl: string;
  observedAtUnixMs: number;
  evidenceId?: string;
}>;
```

### Evidence Rules

1. Old invite responses are historical hints unless paired with current self-membership evidence.
2. Old group messages are historical hints.
3. Old persisted group rows are historical hints unless backed by private self-membership state.
4. Room keys prove access to encrypted history, not active membership.
5. Relay publish success is network propagation evidence, not the source of private local intent.
6. Relay publish failure must not erase private local intent.

## Projection Contract

```ts
type CommunitySelfMembershipStatus =
  | "joined"
  | "left"
  | "pending_join"
  | "pending_leave"
  | "historical"
  | "unknown";

type CommunityVisibilityState =
  | "current"
  | "previous"
  | "hidden"
  | "discoverable";

type CommunitySendabilityStatus =
  | "can_send"
  | "blocked_left"
  | "blocked_pending_leave"
  | "blocked_room_key_missing"
  | "blocked_relay_degraded"
  | "blocked_unknown_membership";
```

A derived community row must separate:

1. whether the community is known,
2. whether the current user is a member,
3. whether the community should appear in the current Network list,
4. whether the user can send,
5. whether the current device can decrypt history.

## Private Self-Membership Intent

Leaving a community creates private account-owned truth:

```ts
type CommunitySelfMembershipIntent = Readonly<{
  scope: CommunityRuntimeScope;
  communityId: string;
  groupId: string;
  relayUrl: string;
  status: "joined" | "left" | "pending_join" | "pending_leave";
  decidedAtUnixMs: number;
  localDecisionId: string;
}>;
```

Private intent must be included in encrypted account backup/restore. It is the primary guard against ghost communities after fresh-window or fresh-device login.

## Relay Publish Outbox

Public propagation is a separate reliability concern:

```ts
type CommunityRelayPublishOutboxItem = Readonly<{
  scope: CommunityRuntimeScope;
  localDecisionId: string;
  communityId: string;
  relayUrl: string;
  operation: "join" | "leave" | "disband" | "metadata";
  status: "pending" | "accepted" | "rejected" | "rate_limited";
  attempts: number;
  nextAttemptAtUnixMs?: number;
  lastError?: string;
}>;
```

Relay rate limiting produces `rate_limited` or `pending`, not rollback of private membership state.

## Restore Classification

Restore/import must not execute historical records as live commands.

Restore may produce:

1. descriptors,
2. historical hints,
3. previous-community rows,
4. room-key history,
5. message history,
6. private self-membership intents.

Restore must not produce:

1. live notification events,
2. ringing voice calls,
3. active membership from old messages alone,
4. active membership from old invite acceptance alone,
5. sendability success without current membership and room-key projection.

## Legacy Adapter Plan

### Provider Adapter

The existing group provider becomes a consumer of the runtime projection and exposes compatibility shape to UI while surfaces migrate.

### Ledger Adapter

Membership ledger becomes either:

1. migration input, or
2. serialized private self-membership projection cache.

It must not override private self-membership intent.

### Tombstone Adapter

Tombstones become migration input into private left intent. They are not the durable owner after migration.

### Chat-State Groups Adapter

Persisted groups become descriptor/history hints. They must not create current membership by themselves.

### Window Event Adapter

Browser-global mutation events become unsupported for new code. Existing events may temporarily feed an adapter only when scoped and diagnosed.

## Implementation Slices

### Slice 1: Contract And Scope Types

Add the explicit scope and projection contract in a shared, runtime-independent module. Existing community code imports the contract before behavior changes.

### Slice 2: Projection Owner Skeleton

Create the runtime owner with pure input-to-projection functions and no UI dependency.

### Slice 3: Restore Hint Migration

Change restore reconstruction to produce historical hints instead of joined ledger entries where evidence is historical.

### Slice 4: Provider Consumption

Change provider hydration to consume runtime projection. The provider may still expose legacy `createdGroups` shape, but the data comes from the owner.

### Slice 5: Leave Intent And Outbox

Add private self-membership intent and relay publish outbox. Wire leave action through this path.

### Slice 6: Legacy Deletion

Remove writes from legacy owner paths after tests and runtime replay are green.

## Required Tests

1. projection unit tests for all truth precedence rules,
2. storage scope tests for account A and B in one process,
3. restore tests proving historical hints cannot resurrect current membership,
4. leave tests proving rate-limited relay publish does not roll back private leave,
5. sendability tests separating membership from room-key availability,
6. provider adapter tests proving UI rows follow projection output.

## Red Flags

Stop implementation and re-evaluate if a slice:

1. adds another mutable source of community truth,
2. reads active profile scope inside storage instead of receiving scope,
3. treats historical restore evidence as live membership,
4. lets room-key presence imply current membership,
5. emits unscoped browser-global community mutation events,
6. marks success without relay/private-state evidence appropriate to the operation.
