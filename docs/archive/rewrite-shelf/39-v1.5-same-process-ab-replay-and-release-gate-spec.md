# 39 v1.5 Same-Process A/B Replay and Release Gate Spec

_Last reviewed: 2026-05-03 (baseline commit 7111e10a)._

## Purpose

This spec defines the mandatory same-process account A/account B replay matrix for v1.5.0. It exists because single-process multi-profile testing exposes hidden globals, active-profile fallbacks, browser-global events, and shared storage drift that separate browser or separate device tests can miss.

## Test Topology

The required topology is:

1. account A and account B operate in one application process,
2. both accounts can create, join, leave, and rejoin communities,
3. both accounts can send and receive community and direct messages,
4. profile/window/device scope changes occur without process restart,
5. account sync and restore can run while both accounts have historical data in storage,
6. relay degradation and rate limiting can be simulated or observed.

## Core Assertion

If account A and account B share a process, no community, message, notification, call, room key, or restore mutation may use ambient active-profile state when explicit scope is available.

## Required Scope Dimensions

Each replay event should capture:

1. profileId,
2. publicKey suffix,
3. deviceId when available,
4. windowId when available,
5. account sync phase,
6. relay runtime phase,
7. operation source,
8. restore/live/static classification.

## Replay Matrix

| ID | Scenario | Steps | Expected outcome | Release gate |
| --- | --- | --- | --- | --- |
| AB-01 | Clean A/B startup | Start A and B in one process with separate profile scopes | Both load only their own community and message state | blocking |
| AB-02 | A creates community | A creates community while B is active in another surface/window | A sees current community; B sees nothing unless invited | blocking |
| AB-03 | B joins A community | A invites B; B accepts; both sync | B becomes joined; A sees B according to mode guarantees | blocking |
| AB-04 | B leaves community | B leaves; reload B; switch A/B; restore B | B does not see community as current | blocking |
| AB-05 | Leave under relay rate limit | B leaves while relay rejects/rate-limits publish | B remains privately left; outbox records pending/rate_limited | blocking |
| AB-06 | Old restore cannot resurrect | B restores backup containing old invite, old createdGroup, and old messages | B may have historical row, but no current membership | blocking |
| AB-07 | A remains member after B leaves | B leaves; A remains in community | A membership and sendability are not affected by B's private leave | blocking |
| AB-08 | Profile switch during hydrate | Trigger group hydrate, then switch active profile before microtask completes | stale hydrate result is discarded; no cross-profile write | blocking |
| AB-09 | Static message restore | Restore historical messages for A or B | no message notification and no unread alert event | blocking |
| AB-10 | Static voice restore | Restore historical voice invite rows | no ringing UI and no active call session | blocking |
| AB-11 | Room key without membership | Restore room key for a left community | decryption/history may be available; sendability is blocked_left | blocking |
| AB-12 | Membership without room key | Restore joined state without room key | row may be visible; sendability is blocked_room_key_missing | blocking |
| AB-13 | Global event quarantine | Emit legacy community window events without scope | event is ignored or diagnosed as unsupported | blocking |
| AB-14 | Relay degraded create/join/leave | Run lifecycle actions with zero writable scoped relays | UI shows pending/degraded state, not unsupported success | blocking |
| AB-15 | Account sync overlap | Start restore for A while B performs community action | operations remain scoped; no shared mutation | blocking |

## Runtime Evidence Packet

Each manual replay packet must include:

1. runtime snapshot,
2. relay runtime snapshot,
3. relay transport journal snapshot,
4. app event digest,
5. cross-device/community sync digest if available,
6. account A visible community rows,
7. account B visible community rows,
8. account A private membership intents,
9. account B private membership intents,
10. publish outbox state.

Preferred capture:

```js
copy(window.obscurM0Triage?.captureJson(300));
```

Fallback capture:

```js
copy(JSON.stringify({
  runtime: window.obscurWindowRuntime?.getSnapshot?.() ?? null,
  relayRuntime: window.obscurRelayRuntime?.getSnapshot?.() ?? null,
  relayJournal: window.obscurRelayTransportJournal?.getSnapshot?.() ?? null,
  digest: window.obscurAppEvents?.getDigest?.(300) ?? null,
  crossDevice: window.obscurAppEvents?.getCrossDeviceSyncDigest?.(400) ?? null,
}, null, 2));
```

## Automated Test Requirements

v1.5.0 implementation must leave behind automated tests for:

1. explicit scope storage keys,
2. A/B same-process community projection isolation,
3. stale hydrate discard after profile switch,
4. historical restore hint classification,
5. private leave intent precedence,
6. relay publish outbox state transitions,
7. static restore notification suppression,
8. static restore call suppression,
9. sendability projection precedence,
10. legacy unscoped event quarantine.

## Blocking Failure Classes

Any of these blocks v1.5.0:

1. B sees A-only community rows without invite/current evidence,
2. A loses community membership when B leaves,
3. B's left community reappears as current after restore,
4. historical messages notify during restore,
5. historical voice invites ring during restore,
6. a room key makes a left community sendable,
7. legacy window events mutate the wrong account,
8. profile switch during hydration writes to the previous account,
9. relay publish failure clears private leave intent,
10. diagnostics cannot identify which account/profile/window performed the mutation.

## Required Implementation Order

1. add failing tests for AB-04, AB-05, AB-06, AB-08, AB-09, and AB-10,
2. introduce explicit scope contracts,
3. route community projection through one owner,
4. convert restore evidence to static hints,
5. add private leave intent and outbox,
6. quarantine legacy events,
7. run the full replay matrix before release candidate tagging.

## Manual Replay Notes

Manual replay should record not just the final UI state but also the evidence chain:

1. what user action happened,
2. what private state changed,
3. what relay publish result occurred,
4. what restore imported,
5. what projection emitted,
6. what UI consumed,
7. what notification/call path did or did not fire.
