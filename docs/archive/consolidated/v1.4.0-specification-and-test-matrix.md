## Direct Implementation Order

The next implementation windows should execute in this order:

### Slice 1: Community Modes and Relay Guarantees

Goal:

1. define sovereign-room vs managed-workspace guarantees from relay capability.

Definition of done:

1. a file-backed guarantee matrix exists,
2. unsupported universal community promises are removed from roadmap truth,
3. future implementation slices can point to an explicit mode contract.

### Slice 2: Community Membership Projection

Goal:

1. converge member truth across network page, community page, and member modal according to the chosen mode guarantees.

Definition of done:

1. no self-only collapse when stronger evidence exists,
2. join/leave updates behave honestly for the active mode,
3. restored joined communities reconstruct with other known members or bounded best-effort outputs according to the mode contract.

### Slice 3: Room-Key and Sendability Convergence

Goal:

1. align membership truth with room-key/sendability truth and future coordination guarantees.

Definition of done:

1. joined state without usable room key becomes explicit,
2. sendability reflects canonical evidence,
3. runtime degraded states do not claim sendability success.

### Slice 4: Restore Import Authority

Goal:

1. stop restore from competing with later projection/provider truth.

Definition of done:

1. restore import produces one canonical packet,
2. direct chat-state restore is bounded and named,
3. restored state is not later demoted silently.

### Slice 5: DM Conversation Authority

Goal:

1. converge sidebar and thread history on one authority.

Definition of done:

1. same restored DM truth in list and timeline,
2. no metadata-only shell with missing history,
3. regression tests ratchet “restore must not thin DM history.”

### Slice 6: Relay/Runtime Protection

Goal:

1. preserve liveness and explicit degraded outcomes under transport churn.

Definition of done:

1. route liveness stays bounded,
2. runtime/relay owners emit sufficient diagnostics,
3. no silent transport-path degradation into optimistic UI truth.

## Required Test Ladder Per Slice

Every `v1.4.0` slice must satisfy this ladder where applicable:

1. contract/type tests,
2. reducer/projection tests,
3. persistence/import tests,
4. provider/adapter integration tests,
5. runtime replay for lifecycle-, relay-, or cross-device-sensitive behavior.

## Required Runtime Evidence Packet for v1.4.0 Completion

Before `v1.4.0` can be declared complete, record runtime evidence for:

1. sovereign-room behavior on public/default relays,
2. managed-workspace behavior on stronger/private relay assumptions when implemented,
3. fresh-window login preserving DM history where community continuity depends on it,
4. fresh-window login preserving the supported community directory/roster guarantee for the active mode,
5. real-time community join/leave convergence for the supported mode,
6. degraded relay/runtime replay without route freeze or silent truth thinning.

## Red-Flag Conditions

Do not mark a row complete if:

1. runtime still depends on page-level fallback reconstruction,
2. tests pass but runtime truth is thinner,
3. a slice adds another compatibility mutation path,
4. a mode promises stronger community guarantees than its relay/runtime contract supports,
5. the fix cannot be resumed from repo docs alone.
