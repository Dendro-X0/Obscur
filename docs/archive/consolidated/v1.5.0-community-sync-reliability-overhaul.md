# v1.5.0 Community Sync Reliability Overhaul

**Status:** Draft for implementation preparation  
**Release Lane:** `v1.5.0`  
**Last Updated (UTC):** 2026-05-03  
**Release posture:** Blocked until community systems, data synchronization, reliability, and multi-profile isolation pass the gates in this document.

## Purpose

This roadmap defines the v1.5.0 refactor/rewrite lane for the community runtime, restore/import behavior, data synchronization, and same-process multi-profile reliability.

The immediate test topology is account A and account B operating in a single process. That topology is mandatory because it exposes hidden globals, active-profile fallbacks, browser-global events, shared storage, and owner drift that separate-browser tests can miss.

## Release Blocker Statement

Do not release v1.5.0 while any of these remain true:

1. a community that a user left can reappear as current membership after login, restore, profile switch, or account sync,
2. restored historical community/message/call evidence can trigger live UI effects,
3. accounts A and B can observe or mutate each other's account/profile-scoped community state in one process,
4. relay publish failure or rate limiting can make local/private leave intent disappear,
5. community membership, descriptor, room-key, and sendability truth have more than one live mutation owner,
6. release claims rely on docs or unit tests without A/B runtime replay evidence.

## Scope

### In Scope

1. community create/join/leave/rejoin lifecycle,
2. community descriptor, roster, self-membership, room-key, and sendability projections,
3. restore/import classification of historical evidence,
4. account/profile/window/device scope contracts,
5. same-process A/B testing and diagnostics,
6. relay publish reliability and retry outbox semantics for membership actions,
7. static restore boundary for messages, calls, notifications, and unread counts,
8. deletion or quarantine of legacy owner paths after replacement.

### Out Of Scope

1. broad feature expansion unrelated to core trust paths,
2. cosmetic community UI changes that do not support the new projection contract,
3. release marketing claims before replay evidence exists,
4. protocol purity work that delays reliable user-owned private state.

## Current Evidence

The current system already has partial replacement pieces:

1. CRDT primitives package exists.
2. Community membership CRDT exists.
3. CRDT sync protocol exists.
4. Community membership recovery emits descriptor and membership projections.
5. Group provider still owns live state mutation and persistence.
6. Restore merge still reconstructs joined membership from historical chat evidence in some paths.
7. Ledger and tombstone stores still depend on ambient profile scope and browser storage.
8. Community mutation still flows through browser-global events.

The conclusion is not "add one more guard." The conclusion is to make one community runtime/projection owner canonical and demote legacy modules to migration inputs or adapters.

## Target Architecture

### Canonical Runtime Owner

v1.5.0 introduces a single community runtime owner. Its implementation name may change during refactor, but the owner contract is fixed:

1. consume all community evidence,
2. classify evidence source and freshness,
3. derive descriptor, self-membership, roster, room-key, and sendability projections,
4. persist private account truth with explicit profile/account/window/device scope,
5. expose one read API to React surfaces,
6. expose one mutation API for create/join/leave/rejoin,
7. emit diagnostics at every projection and mutation boundary.

The existing group provider should become a React adapter, not the authoritative owner.

### Required Scope Contract

Every community, restore, sync, and profile-sensitive operation must receive explicit scope:

```ts
type CommunityRuntimeScope = Readonly<{
  profileId: string;
  publicKeyHex: string;
  deviceId: string;
  windowId: string;
}>;
```

Storage modules must not derive profile scope from mutable ambient state at access time when the caller can pass scope explicitly.

### Required Evidence Contract

```ts
type CommunityEvidenceSource =
  | "local_user_action"
  | "live_relay"
  | "restore_backup"
  | "legacy_migration"
  | "retry_outbox";

type CommunityEvidenceFreshness =
  | "current"
  | "historical"
  | "unknown";
```

Historical evidence may recover descriptors, previous-community rows, known participants, and room-key history. Historical evidence must not create current membership by itself.

### Required Projection Contract

```ts
type CommunitySelfMembershipStatus =
  | "joined"
  | "left"
  | "pending_join"
  | "pending_leave"
  | "historical"
  | "unknown";

type CommunitySendabilityStatus =
  | "can_send"
  | "blocked_left"
  | "blocked_pending_leave"
  | "blocked_room_key_missing"
  | "blocked_relay_degraded"
  | "blocked_unknown_membership";
```

## Truth Precedence

Community self-membership precedence is fixed:

1. private durable self-membership intent,
2. signed current live membership evidence,
3. accepted retry-outbox result,
4. explicit non-joined ledger state from migration,
5. historical hints from restore/import,
6. persisted legacy chat-state fallback.

Explicit `left` or `pending_leave` must suppress historical `joined` reconstruction.

Room-key availability is separate from membership. A room key may support decryption or history display, but it must not imply active membership.

## Milestone Plan

### M0 Baseline Lock

Deliverables:

1. land this roadmap and supporting specs,
2. record all current community/sync/profile drift as release blockers,
3. add failing or pending A/B replay tests for the known failure classes,
4. identify all current mutation owners and mark which will become adapters.

Exit criteria:

1. docs check passes,
2. implementation slices can start from a stable owner map,
3. no stale roadmap claims describe community systems as release-ready.

### M1 Scope And Owner Replacement

Deliverables:

1. introduce explicit runtime scope types,
2. thread scope through community persistence and projection reads,
3. stop new code from using ambient active profile fallbacks,
4. convert group provider into a consumer/adapter boundary.

Exit criteria:

1. A/B single-process scope isolation test fails before implementation and passes after,
2. account A cannot read, refresh, or mutate account B community state without explicit scope,
3. diagnostics include profileId, publicKey suffix, deviceId, and windowId where available.

### M2 Projection And Restore Boundary

Deliverables:

1. replace historical `joined` reconstruction with historical hints,
2. centralize membership/descriptor/room-key/sendability projection,
3. ensure restore/import is static by default,
4. prevent notifications/calls/unread alerts from static import.

Exit criteria:

1. left communities cannot resurrect from old invites, group messages, or createdGroups,
2. historical messages do not notify,
3. historical voice calls do not ring,
4. restore output is a projection input, not a live mutation command.

### M3 Leave Intent And Reliability Outbox

Deliverables:

1. durable private leave intent,
2. relay leave publish retry outbox,
3. rate-limit classification and retry scheduling,
4. user-facing pending propagation state.

Exit criteria:

1. relay rate limiting does not undo local/private leave,
2. fresh-window restore honors private leave,
3. outbox retry emits accepted/rejected/rate-limited diagnostics.

### M4 Legacy Deletion And Anti-Drift

Deliverables:

1. stop writing legacy fallback storage as active truth,
2. quarantine browser-global community mutation events,
3. remove or isolate compatibility bridges,
4. update owner docs and release verification docs.

Exit criteria:

1. one canonical mutation owner remains,
2. all remaining legacy paths are read-only migration inputs or deleted,
3. release verification matrix includes A/B same-process replay evidence.

## Required Test Matrix

| Scenario | Required outcome | Gate type |
| --- | --- | --- |
| A creates community, B joins, both send messages | membership, room key, and message views remain scoped and convergent | automated + runtime |
| B leaves community, relay accepts leave | B no longer sees current community after reload/restore | automated + runtime |
| B leaves community, relay rate-limits leave publish | B remains privately left; outbox shows pending propagation | automated + runtime |
| B restores old backup containing invite/group history | historical evidence does not resurrect current membership | automated |
| A/B switch active profiles in one process | no cross-profile community state leakage | automated + runtime |
| static restore imports historical messages | no desktop notification or unread alert event | automated + runtime |
| static restore imports historical voice invite | no ringing call UI | automated + runtime |
| room key exists but membership is left | sendability is blocked_left, not can_send | automated |
| membership joined but room key missing | sendability is blocked_room_key_missing | automated |
| relay degraded during create/join/leave | user sees explicit degraded/pending state, not optimistic success | runtime |

## Diagnostics Requirements

v1.5.0 implementation must add or preserve diagnostics for:

1. community projection recompute input counts,
2. evidence source and freshness classification,
3. private self-membership decision changes,
4. relay publish outbox state changes,
5. restore static import batches,
6. notification/call suppression reason codes,
7. account/profile/window/device scope mismatch,
8. rejected legacy mutation attempts.

## Release Gates

Before v1.5.0 can be tagged:

1. focused owner tests pass,
2. single-process A/B replay tests pass,
3. typecheck passes,
4. docs check passes,
5. runtime replay evidence is recorded for community, restore, notification, call, and profile-switch flows,
6. current-session handoff records no unresolved severe blocker in this lane,
7. stale v1.4.x community stability claims are either updated or superseded.
