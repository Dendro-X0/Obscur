# v1.4.9 Roadmap

## Overview
**Target Release Date**: 2026-06-15  
**Theme**: "Mobile Delivery & Distribution"

v1.4.8 successfully restored CI builds for mobile platforms, but the artifacts weren't being published to GitHub Releases. v1.4.9 focuses on ensuring mobile installers are actually delivered to users through the release pipeline and website.

---

## Goals

### 1. Mobile Artifact Publishing (Primary)
**Problem**: v1.4.8 CI builds mobile APK/AAB/IPA successfully but doesn't publish them to GitHub Releases.

**Deliverables**:
- [ ] Fix artifact path patterns in `actions/upload-artifact` steps
- [ ] Ensure `actions/download-artifact` in publish job retrieves mobile artifacts
- [ ] Verify mobile assets appear in GitHub Release assets list
- [ ] Add mobile artifact verification step in CI

**Files**:
- `.github/workflows/release.yml`

---

### 2. Mobile Signing Integration
**Problem**: Unsigned mobile builds work for testing but can't be distributed through official stores.

**Deliverables**:
- [ ] Document Android keystore setup process
- [ ] Document Apple Developer certificate requirements
- [ ] Add conditional signing to Android CI (sign when secrets present)
- [ ] Add conditional signing to iOS CI (sign when secrets present)
- [ ] Test signed build pipeline with test certificates

**Secrets Required**:
- `ANDROID_KEYSTORE_PASSWORD`
- `ANDROID_KEYALIAS`
- `ANDROID_KEY_PASSWORD`
- `APPLE_CERTIFICATE_BASE64`
- `APPLE_CERTIFICATE_PASSWORD`
- `APPLE_DEVELOPMENT_TEAM`

---

### 3. Website Download Page Enhancement
**Problem**: v1.4.8 website has mobile section but no actual mobile assets to link to.

**Deliverables**:
- [ ] Update download page to show mobile assets from GitHub Releases API
- [ ] Add QR codes for mobile downloads
- [ ] Add "Download for Android" / "Download for iOS" direct links
- [ ] Handle signed vs unsigned build distinction in UI

**Files**:
- `apps/website/src/app/download/page.tsx`
- `apps/website/src/app/site-content.ts`

---

### 4. App Store Distribution Preparation
**Problem**: Mobile builds exist but aren't ready for store submission.

**Deliverables**:
- [ ] Prepare Android Play Store listing assets
- [ ] Prepare iOS App Store listing assets
- [ ] Create `fastlane` configuration for automated store deployment
- [ ] Document store submission process

**New Files**:
- `fastlane/Appfile`
- `fastlane/Fastfile`

## Technical Requirements

### CI Pipeline Changes
```yaml
# Verify mobile artifacts are published
check-mobile-artifacts:
  needs: [publish-release]
  steps:
    - name: Verify APK exists in release
      run: gh release view ${{ github.ref_name }} --json assets | jq '.assets[] | select(.name | contains(".apk"))'
    - name: Verify AAB exists in release
      run: gh release view ${{ github.ref_name }} --json assets | jq '.assets[] | select(.name | contains(".aab"))'
```

### Artifact Path Mapping
| Platform | Build Output | Upload Pattern | Release Asset |
|----------|-------------|----------------|---------------|
| Android | `app/build/outputs/apk/**/*.apk` | `**/*.apk` | `app-universal-release-unsigned.apk` |
| Android | `app/build/outputs/bundle/**/*.aab` | `**/*.aab` | `app-universal-release.aab` |
| iOS | `gen/apple/build/arm64/*.ipa` | `**/*.ipa` | `Obscur_1.4.9_arm64.ipa` |

## Open Questions

1. **Artifact naming**: Should we include version in mobile artifact names like desktop builds?
2. **Signed vs unsigned**: Should we publish both signed and unsigned builds, or only signed when available?
3. **iOS distribution**: Simulator builds can't be distributed to devices - do we skip iOS builds when signing unavailable?
4. **Minimum Android version**: What API level should we target for Play Store compliance?
