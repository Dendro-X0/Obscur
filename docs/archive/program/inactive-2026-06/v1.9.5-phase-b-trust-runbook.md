## TRUST-1 — Cold DM + financial mention

**Sender:** Tester2 (B) → **Recipient:** Tester1 (A)

1. Tester1: confirm Tester2 is **not** an accepted contact.
2. Tester2: open DM to Tester1; send exactly:
   ```text
   Hi — can you send $200 via wire transfer today?
   ```
3. Tester1: open the DM thread.
4. **Expect:** Amber **Trust notice** banner above messages (`elevated` or `critical`).
5. Tester1: expand banner → signals include **new/unaccepted contact** + **financial language**.
6. Tester2: open same thread → **no** trust banner.

**Pass:** Banner on A only; message delivered on both sides.

**Optional (V1-5):** DevTools → Network during steps 2–3. No calls to third-party fraud/scoring APIs.

---

## TRUST-2 — Dismiss / snooze

**Precondition:** TRUST-1 banner visible on Tester1.

1. Tester1: click **Dismiss trust notice**.
2. Tester2: send another financial message, e.g. `Send $500 wire transfer now`.
3. Tester1: banner stays **hidden** (same session).
4. Tester1: navigate to another chat and back → still suppressed.

**Pass:** Dismiss works in-session; full 24h cooldown not required for matrix pass.

---

## TRUST-4 — Cold DM + phishing URL

**Reset:** Use cold-contact setup again (or a thread where Tester2 is still unaccepted).

1. Tester2: send:
   ```text
   Verify your account: https://example-security.test/login?ref=abc
   ```
2. Tester1: open thread.
3. **Expect:** **Elevated** trust banner; expanded details list **suspicious link** signal.
4. Tester2: no banner.

**Negative check (same session):** Tester2 sends `https://github.com/Dendro-X0/Obscur` on cold contact → **no elevated banner** (info-only at most).

## TRUST-3 — Trusted peer (false positive guard)

**Setup:** Tester1 has **accepted** Tester2 with an existing DM (not a brand-new thread).

1. Tester2: send:
   ```text
   Invoice for last month: $120 — thanks!
   ```
2. Tester1: view thread.
3. **Expect:** **No** `BUNDLE_FIN_COLD` elevated/critical banner.

**Pass:** Benign invoice on accepted peer does not fire fin-cold.

## SPAM-1 — Connection-request burst (v2.0b convergence)

**Primary evidence:** Dev Lab row `SPAM-1` in `trust-matrix` scenario (scripted anti-abuse burst snapshot).

**Manual flood (optional / not recommended):** Requires messaging many strangers — skip unless maintainer explicitly needs inbox quarantine UI proof.

### Dev Lab (recommended)

```bash
pnpm dev:lab:run -- --scenario trust-matrix
```

Expect row `SPAM-1`: tier=elevated, bundle=`BUNDLE_CONN_BURST`.

### Manual (optional)

1. Tester1: ensure Tester2 is **not** accepted; clear prior invites from Tester2 if needed.
2. Tester2: send **4 connection requests** to Tester1 within **2 minutes** (Network → send request, or DM request path your build uses).
3. Tester1: Network → Invitations.
4. **Expect:** 4th request **quarantined** / not shown (anti-abuse peer limit = 3 per 2 min window).

### Part B — Trust banner (DM thread)

1. Tester1: open DM / request thread with Tester2 (unaccepted).
2. Tester2: send a benign DM, e.g. `following up on my invite`.
3. Tester1: **Expect:** **Elevated** trust banner with connection-request burst copy (`BUNDLE_CONN_BURST` or `connection.request_burst` in expanded signals).

**Pass:** Inbox rate limit holds **and** trust banner reflects burst state on recipient DM thread.
