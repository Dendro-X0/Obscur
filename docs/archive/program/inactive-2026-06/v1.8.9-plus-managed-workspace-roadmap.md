## Context

[v1.8.8](./v1.8.8-scope.md) closes the **managed-workspace manual lane**: coordination + operator relay + two-client invite → accept → sealed group chat on local infra. That path is **local-first per device**; a VPS or private relay improves **operator control**, not automatic cross-client SQLite parity.

These items are **honest next steps** for teams running **private infrastructure** (coordination worker + `nostr-rs-relay` or equivalent). They are **not** claims about arbitrary public Nostr clients.

## Theme 1 — End-to-end deletion (operator relay)

### Problem

On open Nostr, events are **append-only**. “Delete for everyone” is at best **cooperative redaction** — see [cooperative-redaction-future.md](../messaging/cooperative-redaction-future.md) and [deletion-roster-limitations.md](../messaging/deletion-roster-limitations.md).

When all community traffic flows through an **operator-controlled relay**, that relay can act as a **real-time repository** with policies stricter than the public network.

### What exists today (v1.8.8 baseline)

| Layer | Owner / path | Behavior |
|-------|----------------|----------|
| Publish deletion | `use-sealed-community` → `GroupService.hideMessage` | Signed hide event to community relay scope |
| Local suppression | Tombstones, Tauri `dbInsertGroupTombstone`, chat-state | Hide on this device / viewer gate |
| UI | Chat message menu | “Hide on this device”; group delete path mixes relay + local |
| Relay config | `infra/nostr/nostr-rs-relay.toml` | Dev open relay; **no** server-side purge yet |

### Target architecture (v1.8.9+)

```text
Client (Obscur)  →  signed hide/delete event  →  Operator relay
                                                      ├─ suppress in REQ/EVENT filters (live)
                                                      └─ optional compaction job (stored events)
Coordination (optional)  →  audit / steward policy / webhook hooks
```

| Phase | Deliverable | Exit criteria |
|-------|-------------|---------------|
| **D1** | Relay **filter suppress** for community-scoped hide events | New messages after hide not served; existing clients on Obscur honor tombstones |
| **D2** | **Compaction** job on operator relay (group tag scope) | Maintainer doc + opt-in config; honest UX copy about off-relay copies |
| **D3** | **Strict workspace mode** flag when relay URL is operator-trusted | UI: “Removed from this workspace”; no false “deleted everywhere” |

### Explicit non-goals

- Erasure on relays or clients the operator does not control.
- Parity with Telegram/Discord centralized delete without naming the managed-relay boundary.

## Theme 2 — Group bots (managed communities)

### Problem

Human communities on Telegram/Discord rely on **bots** for announcements, moderation hooks, and integrations. On sealed managed workspaces, bots should be **first-class members** of the same relay + coordination plane — not a separate proprietary API.

### Target architecture (v1.8.9+)

```text
Bot runner (VPS / sidecar)
  ├─ subscribe: same sealed group filters as useSealedCommunity
  ├─ publish: GroupService + community relay scope (dedicated bot npub)
  └─ optional: coordination webhooks / slash-command registry

Human clients  ←→  Operator relay  ←→  Bot runner
Coordination   ←→  permissions, communityId, steward allowlist
```

| Phase | Deliverable | Exit criteria |
|-------|-------------|---------------|
| **B1** | **Outbound-only bot** — announcements via sealed publish | Bot npub listed in descriptor; messages visible in group chat |
| **B2** | **Inbound triggers** — keyword, mention, schedule | Documented rate limits; steward can disable bot |
| **B3** | **HTTP webhooks** via coordination | External systems (alerts, CI) → bot message; audit log |

### Safety defaults

- Bot pubkey in `stewardPubkeys` / explicit `botPubkeys` descriptor field (TBD contract).
- No admin/expel actions unless explicitly granted.
- Rate limits and allowlisted actions per `communityId`.

### Dependency

Requires stable **v1.8.8 managed-workspace** path (invite, join, sealed chat, local history hydration). See [v1.8.8 demo matrix](../assets/demo/v1.8.8/README.md).

## Known limitations carried from v1.8.8 (document, don’t hide)

| Topic | Truth |
|-------|--------|
| Member count in chat header | May lag widen-only roster until terminal/CRDT catches up; terminal cache event improves reactivity |
| History per device | Local hydration on open; each client persists its own copy — VPS relay does not replace per-device store |
| Leave / expel | Terminal cache + participants modal; header uses active CRDT minus terminal lists |
| Public relay federation | Out of scope for managed-workspace lane — see [community-membership-relay-feasibility.md](./community-membership-relay-feasibility.md) |
