# v0.8.2 Ticketized Roadmap

_Last reviewed: 2026-03-04 (baseline commit b8cc5fe)._

_Drafted: 2026-03-04._

This roadmap defines the `v0.8.2` release, focusing on critical fixes for data loss and stability, along with infrastructure improvements deferred from previous cycles.

## 1) Release Theme

1. **Zero Data Loss**: Resolve edge cases where messages or contacts disappear after refresh/restart.
2. **Infrastructure Resilience**: Re-enable high-performance crypto workers and implement real relay-based invite propagation.
3. **UX Smoothness**: Finalize the "Phase 1" performance work with measurable smoothness and responsiveness.

## 2) Scope and Guardrails

### In scope

1. Critical normalization fixes for stable `conversationId` and `communityId` derivation.
2. Re-enabling Web Workers in `CryptoService` with stable Comlink proxies.
3. Functional Nostr-relay propagation for connection requests (moving beyond local-only storage).
4. Permanent fix for group message disappearance under specific relay-url conditions.
5. Standardizing relay-url normalization across all protocol and persistence layers.

### Out of scope

1. New major feature surfaces (Status, Stories, etc.).
2. Complete auth system redesign (Phase 5).
3. Cloud sync or centralized backup.

## 3) Ticket Backlog

### A. Critical Stability & Data Loss

1. `V082-REL-001` Unified Relay Normalization (P0, S)
   - Goal: Ensure `relayUrl` is normalized identically across `community-identity.ts` and `use-sealed-community.ts`.
   - Deliverables: Shared normalization utility used by both `communityId` and `conversationId` derivation.
   - Acceptance: Trailing slashes and case differences do not break message hydration after refresh.

2. `V082-REL-002` Group Message Persistence Audit (P0, M)
   - Goal: Fix "messages vanish after refresh" for Sealed Communities.
   - Deliverables: Hardened `MessagePersistenceService` and `useConversationMessages` logic for group IDs.
   - Acceptance: Group messages remain reliably visible after multiple page refreshes and app restarts.

3. `V082-REL-003` Contact Handshake Reliability (P0, M)
   - Goal: Ensure contacts accepted on one device/session are reliably synced and persisted.
   - Deliverables: Improved `InviteManager` handshake logic and persistent `connectionStore` hydration.

### B. Infrastructure Hardening

1. `V082-INF-001` Crypto Worker Resurrection (P0, M)
   - Goal: Re-enable Web Workers for crypto operations without "phantom" proxy failures.
   - Deliverables: Stable Comlink-based `CryptoService` factory that handles worker lifecycle and main-thread fallback.
   - Acceptance: Heavy crypto (E2EE/NIP-04) does not block the UI thread on browser runtimes.

2. `V082-INF-002` Real Connection-Request Propagation (P0, M)
   - Goal: Send actual Nostr events (Kind 1) or GIFT_WRAPPED events for connection requests.
   - Deliverables: `InviteManager` implementation that publishes to relays instead of just local DB.
   - Acceptance: Connection requests arrive on recipient devices via relays.

### C. Performance & UI Polish

1. `V082-PERF-001` Scroll Match & RAF Synchronization (P1, S)
   - Goal: Eliminate minor jitter when incoming bursts occur during active scroll.
   - Deliverables: Synchronized batch flush with scroll-end heuristics.

2. `V082-PERF-002` Performance Mode Metrics Dashboard (P1, M)
   - Goal: Surface the internal `performanceMonitor` metrics in a debug overlay for maintainers.

## 4) Suggested Sequencing

1. **Wave 1**: `V082-REL-001`, `V082-REL-002`, `V082-REL-003` (Stability First).
2. **Wave 2**: `V082-INF-001`, `V082-INF-002` (Infrastructure).
3. **Wave 3**: `V082-PERF-001`, `V082-PERF-002` (Polish).
4. **Wave 4**: Final regression sweep and release tag.

## 5) Exit Criteria for v0.8.2

1. All P0 tickets complete.
2. Group messages and contacts persist reliably across Refresh/Restart on both PWA and Desktop.
3. `CryptoService` runs in a Worker on capable browsers.
4. Connection requests are dispatched to Nostr relays.
5. All tests green.

## 6) v0.8.2 Go/No-Go Criteria

1. No open P0 bugs.
2. `pnpm version:check` passes.
3. Manual smoke test of "Invite -> Accept -> Refresh -> Message -> Refresh" flow is perfect.
